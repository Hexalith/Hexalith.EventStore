## 1. Epic Summary

**Goal.** Publish persisted events as tenant- and domain-scoped CloudEvents through DAPR pub/sub, keep command processing available during pub/sub outages, drain unpublished backlogs on recovery, and reject overloaded aggregate command queues with per-aggregate backpressure.

**Scope.**

| Story | Title | Outcome |
|---|---|---|
| 4.1 | CloudEvents Publication & Topic Routing | Verified CloudEvents metadata, topic derivation, pub/sub configuration, never-throw publishing, tracing, logging, and added multi-tenant/domain/boundary topic tests |
| 4.2 | Resilient Publication & Backlog Draining | Verified persist-then-publish, `PublishFailed`, drain records, actor reminders, crash resume, DAPR resiliency, timing configuration, and added drain timing/boundary tests |
| 4.3 | Per-Aggregate Backpressure | Implemented and tested aggregate-scoped rejection, pending-count tracking, drain-success decrement, duplicate bypass, configurable options, structured logging, and HTTP 429 handling |

**Final recorded verification from story records.**

- Story 4.1: Tier 1 = 659 passing; Tier 2 = 1381 passing; release build = 0 warnings, 0 errors.
- Story 4.2: Tier 1 = 659 passing; Tier 2 mocked = 1366 passing; 21 DAPR infrastructure integration failures recorded as pre-existing environment dependency; release build = 0 warnings, 0 errors.
- Story 4.3: story file does not record final task notes or test run output; implementation and test files exist in the codebase, including `BackpressureTests.cs`, `BackpressureOptionsTests.cs`, `BackpressureExceptionHandler.cs`, and actor backpressure logic.

## 3. What Went Well

1. **CloudEvents and topic routing became explicit.** Story 4.1 tied DAPR publication to `cloudevent.type`, `source`, `id`, configurable pub/sub name, and `{tenant}.{domain}.events`.

2. **Tenant/domain topic tests closed a real isolation gap.** The new Story 4.1 tests proved different tenants and domains produce distinct topics through the publisher flow.

3. **The publish path preserves command availability.** Story 4.2 verified ADR-P2 ordering: events are persisted before publication, and publication failure produces drain work instead of blocking command acceptance.

4. **Drain recovery has a concrete state model.** `UnpublishedEventsRecord`, reminder naming, retry count, advisory status updates, orphaned reminder handling, and crash resume were all inspected and tested.

5. **Backpressure is isolated at the aggregate boundary.** Story 4.3 implementation rejects overloaded aggregates without making the whole tenant or system unavailable.

6. **Operational contracts are visible to API consumers.** Backpressure maps to HTTP 429 with `Retry-After` and ProblemDetails extensions, matching the existing error-response direction from Epic 3.

---

## 4. What Struggled

1. **Prior retro critical items were not closed before Epic 4.** R3-A1 replay ULID validation, R3-A7 live command-surface verification, and older AggregateType / terminal-status cleanup remain backlog or visibly unresolved.

2. **Story status drift continued.** `sprint-status.yaml` marks `4-2-resilient-publication-and-backlog-draining` done, but the story file still says `Status: review`.

3. **Story 4.3 has thin implementation documentation.** Its story file contains the story and acceptance criteria, but not the task checklist, dev agent record, file list, change log, or final verification summary.

4. **Runtime verification remains uneven.** Story 4.1 deferred the command-to-subscriber pub/sub test to Tier 3; Story 4.2 records DAPR infrastructure integration failures as pre-existing; the retro did not observe a running Aspire topology.

5. **Drain integrity has a documented ambiguity.** Story 4.2 notes that missing events in a drain sequence range are simply not returned, so a successful drain could publish fewer events than the record's `EventCount` if state is inconsistent.

6. **Old story numbering remains in source comments.** Story 4.2 notes XML comments referencing old Story 4.4 / 4.5 numbering, which makes current traceability harder.

## 7. Next Epic Preview - Epic 5

Epic 5 is **Security & Multi-Tenant Isolation**. It builds directly on Epic 4's tenant-domain addressing model:

1. JWT claims and command authorization must align with the tenant/domain values that generate actor IDs, storage keys, and pub/sub topics.
2. Pub/sub topic isolation becomes a security boundary, not just a distribution concern.
3. DAPR access control policies must allow the required EventStore call graph while blocking unauthorized service interactions.
4. Keycloak-backed E2E security tests need the running topology to be trustworthy.

Winston (Architect): "Epic 5's plan is still directionally sound, but it should start by treating Epic 4's routing rules as security invariants. Topic naming, storage keys, and actor identity must agree everywhere."

Alice (Product Owner): "The business value is tenant confidence. If we say isolation is enforced at three layers, the evidence must cover all three layers with real runtime checks."

**Verdict.** Epic 4 succeeded as an event-distribution milestone. The core architecture is coherent: persisted events publish through scoped CloudEvents topics, pub/sub failures produce recoverable drain work, and overloaded aggregates can be rejected without broad outage. The main risk is not the design; it is evidence hygiene and runtime verification. Epic 5 should start only after the high-priority recovery and live-verification actions are visible.

---

## 10. Key Takeaways

1. **Pub/sub routing is now part of the security model.** Tenant-domain topic names must be treated with the same rigor as actor IDs and storage keys.

2. **Verification stories still need complete story records.** If the final task evidence is missing, future agents inherit uncertainty even when code exists.

3. **Resilience claims need runtime proof.** Mocked DAPR calls prove code behavior, but not sidecar policy, broker delivery, subscriber receipt, or reminder execution.

4. **Recovery paths depend on older cleanup.** Replay validation and live command-surface verification remain critical for proving operational recovery end to end.

5. **At-least-once delivery must be paired with explicit data-integrity checks.** Drain should not accidentally turn inconsistent state into apparent success.
