## Story

As an **API consumer**,
I want a running CommandApi host with a POST `/api/v1/commands` endpoint that accepts a command payload and returns `202 Accepted` with a correlation ID,
So that I can submit commands and receive immediate acknowledgment.

## Acceptance Criteria

1. **POST /api/v1/commands endpoint** - Given the CommandApi is running, When I POST a valid JSON (`application/json`) command payload to `/api/v1/commands` with tenant, domain, aggregateId, commandType, and payload fields, Then the response is `202 Accepted`.

2. **Correlation ID in response** - The response body contains a `correlationId` (GUID), generated by middleware before the MediatR pipeline.

3. **Location header** - The response includes a `Location` header pointing to the status endpoint `/api/v1/commands/status/{correlationId}`.

4. **Retry-After header** - The response includes a `Retry-After: 1` header.

5. **Endpoint routing** - The endpoint uses ASP.NET Core controllers with proper routing under `/api/v1/commands`.

6. **Invalid request handling** - When required fields are missing or the request body is malformed, the response is `400 Bad Request` as RFC 7807 ProblemDetails with a `correlationId` extension.

## Dev Notes

### Architecture Compliance

**API Path Convention:** All CommandApi endpoints use `/api/v1/` prefix per architecture decision (REST with path prefix versioning).

**Controllers, not Minimal APIs:** The architecture document explicitly defines `Controllers/CommandsController.cs`, `CommandStatusController.cs`, and `ReplayController.cs`. With 9 stories adding endpoints across Epic 2, controllers provide better organization. Do not use minimal APIs.

**MediatR Pipeline:** This story establishes the MediatR pipeline foundation. The full pipeline order is: LoggingBehavior -> ValidationBehavior -> AuthorizationBehavior -> CommandHandler. For Story 2.1, only the basic handler is needed; LoggingBehavior and ValidationBehavior will be added in Stories 2.2 and 2.3.

**DI Registration Order (Architecture Pattern):**
```csharp
// In CommandApi Program.cs
builder.AddServiceDefaults();           // Aspire: resilience, telemetry, health
builder.Services.AddProblemDetails();   // RFC 7807 framework error formatting
builder.Services.AddAuthentication();   // JWT (stub for now, Story 2.4)
builder.Services.AddAuthorization();    // Policies (stub for now, Story 2.5)
// builder.Services.AddRateLimiter();  // Story 2.9
builder.Services.AddMediatR();          // Pipeline
builder.Services.AddControllers();      // MVC controllers
// builder.Services.AddEventStoreServer(); // Story 3.1+
// builder.Services.AddActors();       // Story 3.1+
```

**Error Response Format:** All errors must use RFC 7807 ProblemDetails (D5). Call `AddProblemDetails()` so framework-level errors (model binding failures, 415 Unsupported Media Type, etc.) are automatically formatted as ProblemDetails. Add a basic `IExceptionHandler` to catch unhandled exceptions and return ProblemDetails without stack traces (rule #13). Story 2.2 will add the full global exception handler with `correlationId`, `tenantId`, and `validationErrors` extensions.

**Enforcement Rules to Follow:**
- Rule #7: ProblemDetails for all API error responses
- Rule #9: correlationId in every structured log entry
- Rule #10: Register services via `Add*` extension methods — create `AddCommandApi()` extension
- Rule #12: Command status writes are advisory (status writing comes in Story 2.6, but the correlationId generation pattern starts here)
- Rule #13: No stack traces in production error responses

### Critical Design Decisions

**CorrelationId Generation Location:** CorrelationId MUST be generated in `CorrelationIdMiddleware`, NOT in the MediatR handler or controller. The middleware runs before everything else, ensuring the correlationId is available for ALL structured log entries — including validation failures and auth failures that never reach the handler. The handler READS the correlationId from `HttpContext.Items`, it does not generate its own.

**Request vs Domain Model Separation:** `SubmitCommandRequest` (in CommandApi/Models/) is an API-facing DTO with `JsonElement` payload — it is deliberately separate from `CommandEnvelope` (in Contracts). They serve different purposes: the request model handles JSON deserialization from HTTP clients, while `CommandEnvelope` is the internal domain type with `byte[]` payload. The handler maps between them. Do NOT use `CommandEnvelope` as the API request model.

**Payload Type:** The `payload` field in `SubmitCommandRequest` should be typed as `JsonElement` to accept any valid JSON value (object, array, string, number) without imposing schema. The handler will serialize it to `byte[]` when creating a `CommandEnvelope` (in a future story).

**Unauthenticated Endpoint Warning:** This endpoint has NO authentication until Story 2.4 (JWT Authentication). It MUST NOT be deployed to any shared or production environment before Story 2.4 is complete. Local development only.
