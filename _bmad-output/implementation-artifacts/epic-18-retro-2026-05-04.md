### Challenges and Friction

- MCP protocol hygiene mattered immediately. Story 18.1 correctly reserved stdout for JSON-RPC and pushed logging to stderr, but this rule has to stay visible because any `Console.Write*` in tool code can corrupt the transport.
- SDK and .NET API details created implementation friction: tool class accessibility, `AddHttpClient<T>` package requirements, `GetFromJsonAsync` null/empty-body behavior, and HTTP status mapping all needed adjustment.
- Error semantics grew organically. The epic had to standardize `unauthorized`, `not-found`, `server-error`, `unreachable`, `timeout`, `invalid-operation`, `invalid-input`, and `service-unavailable` as distinct operational signals.
- Approval gates are useful but client-mediated. The server can require `confirm: true`, but it cannot prove that the MCP client presented the preview to a human before execution.
- Session context improved ergonomics but required careful scope limits. The epic correctly limited auto-fill to list/overview tools and kept detail, diagnostic, and write tools explicit.
- Runtime evidence is not as strong as unit evidence. The story records show strong builds and tests, but the later traceability report marks NFR43 p99 latency as partial rather than fully proven.

### Breakthrough Patterns

- `ToolHelper` became the shared contract for JSON serialization, validation, previews, and error mapping.
- Partial `AdminApiClient` files kept read, diagnostic, write, tenant, and storage concerns separated while preserving a single configured HTTP client.
- Preview-first write tools created a consistent safety shape for AI-driven remediation.
- `InvestigationSession` gave MCP workflows memory without adding persistence or multi-user complexity.
- URI-encoding and null-result handling became explicit test themes instead of incidental behavior.

### Key Takeaways

1. The shared Admin API strategy works: Web UI, CLI, and MCP can remain thin clients over one operational API.
2. MCP needs stricter output discipline than CLI. stdout is protocol, not a logging surface.
3. AI-agent tools need stable error vocabulary as much as they need happy-path data.
4. Approval gates should remain structured, explicit, and visible in every write tool.
5. Session context is valuable only when bounded. Auto-scope list/overview tools; require explicit parameters for targeted reads and writes.
6. Unit coverage is strong, but NFR43 still needs live runtime proof before being claimed as fully satisfied.
