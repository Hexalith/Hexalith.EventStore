## Deferred-Work Governance

DW6 adds a small parseable convention while preserving the historical review text below. New entries SHOULD use a canonical marker at the start of the bullet, immediately after the Markdown dash and before the finding title:

`- **[OPEN] Short finding title** - owner: <role-or-person>; next-review-date: YYYY-MM-DD; grouping: <story-key|post-epic-bucket|epic|component|needs-triage>; rationale: <short reason>; evidence: <path-or-link>.`

Canonical markers are case-sensitive. The accepted values are:

- `OPEN` - unresolved and not yet routed. Requires `owner:`, `next-review-date: YYYY-MM-DD`, `grouping:`, and a short rationale or evidence link.
- `STORY:<id>` - follow-up is owned by an existing or proposed story key in repository story-key style, such as `post-epic-deferred-dw6-deferred-work-governance`. Requires `owner:`, `next-review-date: YYYY-MM-DD`, `grouping:`, and rationale or evidence. DW6 reports unknown story ids as advisory until product or architecture chooses to make existence validation blocking.
- `ACCEPTED-DEBT` - consciously retained risk. Requires rationale and should include `owner:` plus `next-review-date:` unless the entry explains why the debt is intentionally unscheduled.
- `RESOLVED` - closed by completed work, evidence, or a documented correction. Owner and next-review-date are optional.
- `DUPLICATE` - closed because another entry or story owns the same issue. Include the owning item when practical.
- `NO-ACTION` - closed because the finding is obsolete, intentionally not actionable, or outside the product boundary. Include the reason.

Recommended grouping values include a story key, post-epic bucket, epic, component, or `needs-triage`. Use `needs-triage` only when sprint planning still needs to decide whether to promote the item to a story, accept it as debt, mark it duplicate, or close it as no-action.

Reviewer handoff: when a code review defers work, append one canonical bullet under the relevant "Deferred from" heading. Keep the original review wording that explains the risk, then add the marker and metadata instead of rewriting the finding into a new style.

Retrospective handoff: each retrospective should run the checker and summarize the current `OPEN` count, `STORY` promotions, and any `needs-triage` groups that should become sprint backlog candidates.

Legacy compatibility is deliberate. The checker recognizes or reports these forms as legacy-advisory rather than forcing a full rewrite: `STORY:`, `STORY:<id> / ACCEPTED-DEBT`, `DW1 disposition`, `DW2 disposition`, `DW3 disposition`, `DW4 disposition`, `DW5 disposition`, `RESOLVED-IN-`, checkmark-prefixed resolved lines, and free-text notes documented as legacy-advisory. When multiple markers appear on one entry, the deterministic primary-disposition precedence is `RESOLVED`, `DUPLICATE`, `NO-ACTION`, `STORY`, `ACCEPTED-DEBT`, then `OPEN`; secondary markers are compatibility context.

Checker command:

```powershell
.\scripts\check-deferred-work.ps1 _bmad-output/implementation-artifacts/deferred-work.md
```

Use `--json` for stable machine-readable output and `--legacy-advisory` when running against the full historical ledger. Exit code `1` is blocking for malformed canonical `OPEN` or `STORY:<id>` entries, missing owner/date/grouping metadata on those entries, and unclassified live bullets. Legacy unclassified bullets, unknown `STORY:<id>` existence checks, and mixed-marker compatibility notes are advisory. The checker is read-only by default; it reports redacted and capped excerpts but never rewrites the ledger.
