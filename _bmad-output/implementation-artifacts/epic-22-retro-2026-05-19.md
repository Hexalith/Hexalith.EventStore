## 1. Epic Summary

**Outcome.** Epic 22 turned EventStore into the stable gateway contract for downstream bounded contexts. The planned objective was to let downstream services integrate through public packages and platform-owned policies instead of copying wire DTOs, referencing runtime internals, or guessing query/replay/publishing/protection behavior.

- Stream read/replay APIs and projection rebuild checkpoints.
- Payload/snapshot protection, unreadable protected data behavior, crypto-shredding workflow, restored-backup safety, and protected-data redaction across operational surfaces.

## 2. What Went Well

1. **Package ownership became concrete.** Story 22.1 closed the Contracts/Client/Testing boundary with DTO JSON shape tests, gateway client behavior, deterministic fake/builders, package docs, and generated API docs. The runtime/server assemblies are now clearly described as non-consumer surfaces.

2. **Projection query serving stopped leaking Server internals.** Story 22.2 moved the public adapter model into downstream-usable contracts, pinned actor routing semantics, and added package-dependency proof so query fakes and adopter examples do not reinforce `Hexalith.EventStore.Server` coupling.

3. **Gateway authorization moved to the right owner.** Story 22.3 made EventStore responsible for tenant lifecycle, membership, role, and permission checks before command/query execution. It added stable reason codes and fail-closed behavior across runtime, validation, client, and test surfaces.

4. **Query behavior became a public contract.** Story 22.4 added additive query policy types, metadata, paging defaults, stable reason codes, ETag safety rules, and client/fake parity. This made machines branch on status/type/reason metadata instead of mutable text.

5. **Publishing claims became more honest.** Story 22.5 explicitly separated durable publish-after-persist, drain/dead-letter behavior, backend configuration, and proof levels. The work resisted overstating RabbitMQ/Kafka/Service Bus guarantees as "proven" when only configuration and manual proof paths existed.

6. **Replay and rebuild got real operational semantics.** Story 22.6 added public stream read/replay DTOs, gateway client support, fake helpers, projection rebuild checkpoints, operator lifecycle endpoints, stable reason codes, and many concurrency/idempotency hardening patches.

7. **Protected-data handling became systemic.** Stories 22.7a through 22.7d moved from metadata hooks to fail-closed unreadable behavior, crypto-shredding/restored-backup safety, and redaction across logs, ProblemDetails, Admin API/UI, CLI, MCP, replay, rebuild, backup, and test artifacts.

8. **A real runtime defect was closed with runtime evidence.** R22A1 fixed the `QueryRouter` typed-proxy-to-weak-proxy DAPR bug that broke `list-tenants`. The final evidence includes focused Server tests, full Release build, `eventstore-admin` Tenants smoke returning `200 OK` with `[]`, and no fresh QueryRouter weak-proxy NRE entries.

## 3. What Struggled

1. **Story scope repeatedly expanded under review pressure.** Story 22.6 in particular accumulated many review passes, decisions, patches, carries, and defers. The work produced value, but the story artifact became both implementation plan and audit ledger.

2. **Runtime proof was inconsistent across environments.** Some stories had Aspire/DAPR evidence; others recorded missing DAPR CLI, no active AppHost, Aspire version mismatch, or environment-gated integration proof. The latest R22A1 smoke was strong, but 22.5 backend proof and parts of 22.6 remain bounded by environment availability.

3. **Container rows needed sharper operational rules.** The epic handled this by marking parent/container rows as unassignable and splitting 22.7d into children, but the pattern arrived after readiness friction. The sprint-status comments now encode the rule; future epics should start with that clarity when scope is obviously cross-surface.

4. **Review found several "tests pass but proof is weak" cases.** Examples include tests that did not exercise production routing, positive-only package-boundary proof, missing negative assertions, and coverage that did not fail against the old implementation. R22A1 corrected this pattern with an old-implementation-sensitive guard.

5. **Documentation and generated API docs were valuable but fragile.** Some generated docs were refreshed cleanly; others were blocked by pre-existing XML documentation warnings, long filename/signature behavior, or missing tool restore prerequisites. The team documented those blockers instead of pretending the refresh was complete.

6. **Sprint tracking lagged reality.** All Epic 22 story rows were done and R22A1 was done, but `epic-22` was still `in-progress` and `epic-22-retrospective` was still `optional` before this retrospective update. Epic 21 had the same tracking-drift lesson.

## 4. Recurring Review Feedback Themes

- **Do not overstate proof.** Mark Redis/Aspire smoke, unit proof, configured backend support, and manual proof paths distinctly.
- **Use stable machine contracts.** ProblemDetails type URI, status, reason code, warning code, and metadata are the contract; title/detail/free text are not branching APIs.
- **Fail closed on ambiguity.** Tenant/RBAC ambiguity, protected-data unreadability, malformed projection responses, unsafe diagnostics, and restore-admission uncertainty all needed deterministic safe outcomes.
- **Tests must protect the real implementation path.** Several review fixes strengthened tests so they would fail against the previous production bug or missing guard, not merely pass with a friendly seam.
- **No raw protected data in operational surfaces.** The same no-leak principle had to be applied to logs, ProblemDetails, Admin API/UI, CLI, MCP, replay, rebuild, backup validation, and evidence artifacts.
- **Cancellation and retry semantics are correctness behavior.** Query routing, rebuild checkpoint writes, stream reads, and backup/rebuild flows repeatedly needed cancellation-before-classification and retryability tightening.

## 13. Summary

Epic 22 completed a major platform-contract consolidation. It made downstream integration explicit across public packages, projection adapters, gateway authorization, query policy, publishing semantics, stream replay, rebuild checkpoints, and protected-data handling. The strongest pattern was the team's willingness to turn fuzzy behavior into typed contracts, reason codes, docs, fakes, and regression tests.

The main lesson is equally clear: when a story touches public contracts and distributed runtime behavior, proof quality matters as much as implementation. Epic 22 improved that discipline substantially, especially by the time R22A1 landed with a production-path regression guard and live smoke evidence. The next planning move should preserve that standard while paying down the remaining proof and process gaps.
