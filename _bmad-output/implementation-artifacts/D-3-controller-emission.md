# Story D.3: Controller Emission

Status: review

## Story

As a **domain-module author building on Hexalith.EventStore**,
I want **the REST API Roslyn generator to emit compiling typed ASP.NET Core controllers for REST-enabled command and query contracts**,
so that **domain UI hosts can expose OpenAPI-visible, gateway-backed REST endpoints without hand-written controllers or BFF wrappers**.

## Story Context

This is **story D3 of Epic D - REST Controller Source Generator**. D1 created the contract seam (`ICommandContract`, `RestRouteAttribute`, `RestApiAttribute`). D2 creates the analyzer project, discovers command/query marker types, and emits a deterministic manifest. D3 is the first story that turns that discovery model into generated ASP.NET Core controller source.

**Hard dependency:** D3 must start from a completed D2. If D2 is still marked `in-progress`, finish and verify D2 before implementing controller emission. At the time this story was created, the worktree already contained uncommitted D2 files under `src/Hexalith.EventStore.RestApi.Generators/` (`RestApiGenerator`, parsers, descriptors, and manifest emitter), while sprint status still marked D2 as `in-progress`. Treat those as existing work to preserve, not as changes to revert.

D3 must not adopt Sample/Counter or Tenants contracts, delete existing hand-written wrappers, add the D4 persistent generator test project, or register the analyzer in release packaging. Those remain later stories:

- **D4** - generator tests with `CSharpGeneratorDriver` and route/misuse coverage.
- **D5/D6** - in-repo Sample/Counter query and command proof.
- **D7** - Tenants UI host proof in the submodule.
- **D8** - release package inventory, docs, and governance guardrails.

Source of truth: `_bmad-output/planning-artifacts/sprint-change-proposal-2026-06-21.md` Section 4, Epic D row for D3, plus D1/D2 story files and the current gateway client/controller contracts.

## Acceptance Criteria

1. **D2 baseline is preserved and verified before D3 emission work begins.**
   - `RestApiGenerator` still implements `IIncrementalGenerator` and remains analyzer-only (`netstandard2.0`, `IsRoslynComponent=true`, `EnforceExtendedAnalyzerRules=true`).
   - The generator still discovers `ICommandContract`, `IQueryContract`, `[RestApi]`, and optional `[RestRoute]` by metadata-name strings only.
   - The D2 manifest artifact either remains emitted or is deliberately retained behind the same deterministic opt-in path. Do not delete D2 evidence unless D4 or an explicit reviewer later directs it.
   - If any D2 files are absent on the implementation branch, complete D2 first rather than merging D2 skeleton work into D3 without marking that dependency.

2. **Generated controller source is emitted for an opted-in assembly.**
   - When the consuming compilation has `[assembly: RestApi(...)]` and at least one discovered message contract, emit deterministic controller source in addition to the D2 manifest.
   - Generated hint names are stable and collision-free, e.g. `Hexalith.EventStore.RestApi.<sanitized-scope>.Controller.g.cs`.
   - Generated code uses `// <auto-generated />`, `#nullable enable`, file-scoped namespace, and no copyright header.
   - Controller namespace is deterministic: prefer `build_property.RootNamespace` from `AnalyzerConfigOptionsProvider`; fall back to a sanitized compilation assembly name; append `.Generated`.
   - Controller class is `public sealed partial`, derives from `ControllerBase`, and has `[ApiController]`, `[Authorize]`, `[Route("<routePrefix>")]`, `[Tags("<tag-or-domain>")]`, and `[Consumes("application/json")]` where appropriate.
   - The generated constructor injects only `IEventStoreGatewayClient`; do not inject MediatR, DAPR clients, domain services, tenant validators, RBAC validators, projection actors, or state stores.

3. **Route generation honors D1 `RestRouteAttribute` and convention fallback.**
   - For `[RestRoute(verb, template)]`, emit the matching `[HttpGet]`, `[HttpPost]`, `[HttpPut]`, `[HttpPatch]`, or `[HttpDelete]` attribute.
   - Relative templates are emitted relative to the controller route prefix; templates beginning with `~/` are emitted as absolute ASP.NET Core route templates.
   - Empty string templates remain valid and route at the controller prefix root.
   - Without `[RestRoute]`, commands default to `POST` at the controller prefix root.
   - Without `[RestRoute]`, queries default to `GET` at the controller prefix root when their payload can be built from route/query values. If a query requires a body payload shape that the generator cannot derive, emit a diagnostic instead of guessing.
   - Do not reject empty or whitespace route templates at compile time in D3; D1 intentionally allows them. Only report diagnostics for shapes the generated code cannot compile or cannot map deterministically.

4. **Generated command actions delegate through `IEventStoreGatewayClient.SubmitCommandAsync`.**
   - Command actions accept route parameters plus `[FromBody] <CommandType> body` and `CancellationToken`.
   - Null command body returns a 400 `ProblemDetails`.
   - `SubmitCommandRequest` is built with:
     - `MessageId = UniqueIdHelper.GenerateSortableUniqueStringId()`
     - `Tenant = ResolveTenant(...)`
     - `Domain = <CommandType>.Domain`
     - `AggregateId = body.AggregateId`
     - `CommandType = <CommandType>.CommandType`
     - `Payload = JsonSerializer.SerializeToElement(body, generated JsonOptions)`
   - Awaited gateway calls use `.ConfigureAwait(false)`.
   - Successful command submission returns HTTP 202 with the gateway `SubmitCommandResponse` body, `Retry-After: 1`, and a `Location` header pointing to `/api/v1/commands/status/{correlationId}`.
   - Route-vs-body mismatch is fail-closed: when a route parameter unambiguously maps to the command aggregate id or a same-named public command property, compare it to the body value using ordinal string comparison and return 400 on mismatch. Do not rewrite the body from route data.

5. **Generated query actions delegate through `IEventStoreGatewayClient.SubmitQueryAsync`.**
   - Query actions accept route parameters, supported `[FromQuery]` parameters derived from public query contract properties when present, `[FromHeader(Name = "If-None-Match")] string? ifNoneMatch`, and `CancellationToken`.
   - `SubmitQueryRequest` is built with:
     - `Tenant = ResolveTenant(...)`
     - `Domain = <QueryType>.Domain`
     - `QueryType = <QueryType>.QueryType`
     - `ProjectionType = <QueryType>.ProjectionType`
     - `AggregateId` and `EntityId` derived by deterministic route rules below
     - `Payload` only when non-route query parameters exist; otherwise omit or use an empty JSON object consistently with existing gateway validation.
   - Query route mapping rules:
     - A route parameter named `aggregateId` wins for `AggregateId`.
     - If there is exactly one non-tenant route parameter and no `aggregateId`, use it as `AggregateId`.
     - If there is no route parameter, use `"index"` as `AggregateId`.
     - A route parameter named `entityId` wins for `EntityId`; otherwise use the first non-tenant route parameter that is not the aggregate id. If there are multiple ambiguous non-tenant route parameters, emit a diagnostic.
   - If the gateway result is `IsNotModified`, return HTTP 304 and copy the strong ETag header when present.
   - For HTTP 200, copy the strong ETag header when present and return the raw query payload (`result.Payload`) rather than re-wrapping it in `SubmitQueryResponse`.
   - Do not bypass the generic gateway query pipeline, ETag behavior, RBAC, tenant validation, or problem-details mapping.

6. **Generated tenant resolution is explicit and fail-closed.**
   - `RestTenantSource.System` resolves tenant to the literal `"system"`.
   - `RestTenantSource.Route` requires a route parameter named `tenant` or `tenantId`; absence is a compile-time diagnostic.
   - `RestTenantSource.Claims` reads normalized `eventstore:tenant` claims. Exactly one distinct claim resolves the tenant. Zero claims returns 403. More than one distinct claim returns 400 with a clear "tenant is ambiguous" problem.
   - Do not parse bearer tokens, JWT payloads, or raw `tenants` JSON in generated code. Hosts that need JWT claim normalization must use their authentication/claims-transformation setup.

7. **Gateway exceptions and local validation failures map to support-safe HTTP responses.**
   - Catch `EventStoreGatewayException` in generated helpers and return `ProblemDetails` with the gateway status code, title, type, detail, correlation id, tenant id, reason/reasonCode, validation errors, and `Retry-After` header when available.
   - Do not include bearer tokens, decoded JWT payloads, raw command/query payloads, stack traces, cursor internals, ETags beyond header values, or raw gateway metadata in visible problem details.
   - Local generated validation failures (null body, tenant ambiguity, route/body mismatch, unsupported generated route state) return `application/problem+json`.

8. **Generator implementation remains incremental, deterministic, and analyzer-safe.**
   - Do not add references from the analyzer project to `Hexalith.EventStore.Contracts`, `Hexalith.EventStore.Client`, ASP.NET Core, Sample, or Tenants. Generated source may reference those types in the consuming compilation.
   - Do not use runtime reflection, assembly loading, `Type.GetType`, timestamps, absolute paths, random ids, machine names, or nondeterministic ordering.
   - Keep parse/transform models pure and equatable; do not store `ISymbol` or syntax nodes in model records used across incremental pipeline stages.
   - Do not store state on the generator instance.
   - Keep one primary C# type per file in `src/Hexalith.EventStore.RestApi.Generators/`.

9. **Scope stays additive and contained.**
   - No Sample/Counter contract annotations and no `CounterQueryService` removal in D3.
   - No Tenants submodule edits, no `TenantsQueryController` deletion, and no Tenants UI adoption in D3.
   - No D4 test project, no Verify snapshots, and no release package inventory updates in D3.
   - No AppHost, DAPR component, service-defaults, container, or UI/UX changes.
   - No generated `bin/`, `obj/`, or temporary smoke files are committed.

10. **Verification proves generated controllers compile.**
    - `dotnet build src/Hexalith.EventStore.RestApi.Generators/Hexalith.EventStore.RestApi.Generators.csproj --configuration Release` succeeds with zero warnings.
    - `dotnet build Hexalith.EventStore.slnx --configuration Release -p:UseHexalithProjectReferences=false` succeeds.
    - A non-committed smoke consumer build uses `EmitCompilerGeneratedFiles=true` and references:
      - local `Hexalith.EventStore.Contracts`
      - local `Hexalith.EventStore.Client`
      - the generator project as an analyzer
      - ASP.NET Core shared framework or web SDK as needed for generated controller types
    - Smoke source declares `[assembly: RestApi("api/counter", "counter", RestTenantSource.System)]`, one query contract, one command contract with `AggregateId`, and at least one `[RestRoute]` with a route parameter.
    - Smoke evidence includes the generated controller path and confirms generated source contains `[ApiController]`, `[Authorize]`, `IEventStoreGatewayClient`, `SubmitCommandAsync`, `SubmitQueryAsync`, `ConfigureAwait(false)`, and no `Type.GetType`/reflection-based dispatch.

## Dev Notes

### Top Guardrails

1. **Do not bypass the gateway.** Generated controllers are typed BFF facades. They must call `IEventStoreGatewayClient`, which calls the generic `/api/v1/commands` and `/api/v1/queries` gateway. Do not route directly to MediatR, DAPR service invocation, actors, state stores, `DomainQueryDispatcher`, or domain handlers.
2. **D3 is generator emission only.** Do not annotate real Sample or Tenants messages and do not delete existing wrappers/controllers. Generated code is proven by smoke compilation here; product adoption is D5-D7.
3. **The analyzer stays runtime-assembly-free.** The analyzer project cannot reference Contracts, Client, ASP.NET Core, Sample, or Tenants. It can emit source that references those assemblies in the consumer.
4. **Never execute static contract members in the generator.** Generated code should reference `CommandType.Domain`, `CommandType.CommandType`, `QueryType.QueryType`, and `QueryType.ProjectionType` at runtime. The generator may inspect syntax and symbols, but it must not execute property getters.
5. **Tenant resolution must fail closed.** A generated endpoint that cannot determine one tenant must not choose a random tenant claim.

### Generated Code Shape

Generated code should look conceptually like this, adjusted for actual descriptors:

```csharp
// <auto-generated />
#nullable enable

namespace Example.DomainHost.Generated;

using System.Text.Json;
using System.Text.Json.Serialization;

using Hexalith.Commons.UniqueIds;
using Hexalith.EventStore.Client.Gateway;
using Hexalith.EventStore.Contracts.Commands;
using Hexalith.EventStore.Contracts.Queries;

using Microsoft.AspNetCore.Authorization;
using Microsoft.AspNetCore.Mvc;

[ApiController]
[Authorize]
[Route("api/counter")]
[Tags("counter")]
[Consumes("application/json")]
public sealed partial class CounterRestController(IEventStoreGatewayClient gateway) : ControllerBase
{
    private static readonly JsonSerializerOptions JsonOptions = new(JsonSerializerDefaults.Web)
    {
        DefaultIgnoreCondition = JsonIgnoreCondition.WhenWritingNull,
        Converters = { new JsonStringEnumConverter() },
    };

    [HttpPost("{counterId}/increment")]
    [ProducesResponseType(typeof(SubmitCommandResponse), StatusCodes.Status202Accepted)]
    public async Task<IActionResult> IncrementCounterAsync(
        [FromRoute] string counterId,
        [FromBody] IncrementCounter? body,
        CancellationToken cancellationToken)
    {
        if (body is null)
        {
            return CreateProblem(StatusCodes.Status400BadRequest, "Bad Request", "Request body is required.");
        }

        if (!string.Equals(counterId, body.AggregateId, StringComparison.Ordinal))
        {
            return CreateProblem(StatusCodes.Status400BadRequest, "Bad Request", "Route aggregate id does not match the command body.");
        }

        var request = new SubmitCommandRequest(
            UniqueIdHelper.GenerateSortableUniqueStringId(),
            ResolveTenant(),
            IncrementCounter.Domain,
            body.AggregateId,
            IncrementCounter.CommandType,
            JsonSerializer.SerializeToElement(body, JsonOptions));

        try
        {
            SubmitCommandResponse response = await gateway
                .SubmitCommandAsync(request, cancellationToken)
                .ConfigureAwait(false);

            Response.Headers.RetryAfter = "1";
            Response.Headers.Location = "/api/v1/commands/status/" + Uri.EscapeDataString(response.CorrelationId);
            return Accepted(response);
        }
        catch (EventStoreGatewayException ex)
        {
            return MapGatewayException(ex);
        }
    }
}
```

The exact generated source should be emitted by helpers, not copy-pasted ad hoc into `RestApiGenerator`.

### Query Binding Caution

Tenants proves the hard cases: `GetTenant` maps `{tenantId}` to aggregate/entity id, `ListTenants` uses an `"index"` aggregate, and `GetUserTenants` uses absolute route `~/api/users/{userId}/tenants` with `AggregateId = "index"` and `EntityId = userId`. D3 does not need to replace Tenants yet, but it must avoid an inference model that would make D7 impossible. If a route has multiple non-tenant parameters and no explicit `aggregateId`/`entityId`, report a diagnostic and skip that action rather than generating a wrong endpoint.
