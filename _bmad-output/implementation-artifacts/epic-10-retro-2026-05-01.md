## 3. What Went Well

1. **Audit-first delivery stayed effective.** Each story treated existing SignalR implementation as a hypothesis, verified acceptance criteria against actual code, and filled bounded gaps.

2. **Signal-only design stayed intact.** Broadcasts carry `projectionType` and `tenantId`, not projection state or ETag values. That keeps the query endpoint authoritative and reduces leakage risk.

3. **Broadcast ordering was pinned.** Story 10.1 added evidence that ETag regeneration completes before clients are notified to re-query.

4. **Disabled mode became explicit.** NoOp broadcaster behavior and disabled hub mapping were tested, which protects deployments that do not enable SignalR.

5. **Redis startup behavior improved.** Story 10.2 changed backplane configuration so Redis connection failure does not contradict the fail-open design intent.

6. **Single-instance and multi-instance modes are separated.** The default path works without a Redis dependency, while production backplane configuration remains available through config or environment variable.

7. **Client reconnection behavior matured.** Story 10.3 strengthened repeated reconnection, callback preservation, closed-event logging, and retry policy configurability.

8. **Story records captured boundaries clearly.** The team explicitly documented what is proven by focused tests and what still needs live topology evidence.

---

## 4. What Struggled

1. **End-to-end SignalR proof remains incomplete.** The stories prove focused behavior, but no active AppHost was available during this retrospective to verify command -> projection change -> ETag regeneration -> SignalR client refresh.

2. **Redis backplane evidence is mostly structural.** Story 10.2 proves DI registration and failure-mode handling, but not real cross-instance delivery through Redis.

3. **Latency evidence is bounded to controllable dispatch.** The p99 requirement is acknowledged, but the recorded proof excludes Redis network hop, WebSocket delivery, and UI refresh behavior.

4. **Hub group authorization is a known hardening gap.** Story 10.1 notes that a connected client could join any group name. Signal-only payloads reduce impact, but tenant-aware authorization is still the cleaner production posture.

5. **Planning artifacts still contain stale separator wording.** PRD and epics references still use older `{ProjectionType}-{TenantId}` language in places, while implementation and story records use `{ProjectionType}:{TenantId}`.

6. **Reconnect semantics are easy to overclaim.** Automatic group rejoin does not replay missed signals. Clients must re-query on connect or reconnect to establish current state.

7. **Prior retro follow-through remained weak.** Several Epic 9 action items were not closed by Epic 10, especially live topology proof and planning artifact reconciliation.

## 8. Significant Discoveries

No Epic 10 discovery invalidates the Epic 11 plan. The core SignalR direction is sound.

Epic 10 did reveal preparation needs:

- Live topology proof should cover the complete projection notification path, not only hub or client slices.
- Redis backplane confidence should move from DI registration proof to real multi-instance delivery proof.
- Tenant-aware group subscription authorization should be resolved or explicitly accepted as a security decision.
- Planning artifacts should be normalized to colon-scoped group IDs.
- Client guidance must state that reconnect rejoin is not missed-signal replay.

## 11. Key Takeaways

1. **SignalR is correctly positioned as invalidation, not data delivery.** This keeps security and cache correctness centered on the query pipeline.

2. **Focused tests found real gaps without turning into broad rewrites.** Disabled mode, call ordering, DI wiring, and reconnect behavior all improved through targeted checks.

3. **Runtime topology proof is the next confidence boundary.** The epic is complete from a story perspective, but final real-time confidence requires a running system.

4. **Security posture should not rely only on data-free signals.** Tenant-scoped group authorization deserves explicit handling.

5. **Reconnect support needs clear consumer guidance.** Rejoin restores future signals; it does not replay missed projection changes.
