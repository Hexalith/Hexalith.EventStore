### Challenges and Friction

- DAPR metadata is sidecar-local. Actor and subscription data required remote EventStore sidecar metadata calls, which introduced port, startup-order, and access-control diagnostic complexity.
- DAPR SDK typed metadata coverage was incomplete for several needs. Runtime version, subscriptions, and resiliency policies needed fallback paths: extended metadata, HTTP JSON parsing, or YAML file parsing.
- The shared `IDaprInfrastructureQueryService` grew broad. It kept the epic cohesive, but it now owns components, sidecar info, actors, pub/sub, and resiliency parsing.
- UI polling and page lifecycle patterns needed repeated hardening. Several review fixes targeted timer resilience, disposal behavior, cancellation tokens, and error handling.
- Health history introduced the first Admin.Server background collector. This is useful, but it creates new operational concerns around state-store write volume, retention, and multi-instance behavior.
- Several review findings were consciously deferred or skipped, especially remote fallback test depth, interactive UI behavior tests, state-store growth caps, and DAPR internal actor key convention validation.

### Breakthrough Patterns

- `RemoteMetadataStatus` replaced a misleading boolean and gave the UI three concrete states: NotConfigured, Unreachable, and Available.
- DAPR sub-pages became a repeatable pattern: DAPR landing page button, dedicated page, dedicated UI client, Admin API endpoint, shared model, and focused tests.
- The resiliency viewer made YAML the source of truth instead of duplicating DAPR config in appsettings.
- Health history used DAPR state store as a lightweight admin telemetry store without adding a database dependency.
- Review feedback improved product behavior, not just code hygiene: the best fixes changed what an operator would understand during an incident.

### Key Takeaways

1. Sidecar-local APIs need explicit diagnostic states. Binary "available" flags hide the difference between configuration, reachability, and empty data.
2. The Admin DAPR surface is now broad enough that consistency is a product feature. Page patterns, API clients, models, and error wording must stay aligned.
3. Runtime evidence matters more for DAPR features than for ordinary UI CRUD. Metadata calls, actor state conventions, sidecar ports, and access control all need live proof.
4. File-backed DAPR configuration views should read canonical DAPR artifacts instead of mirroring them in application config.
5. Background collection is a new operational responsibility. Once Admin.Server writes telemetry, it needs retention, growth, and multi-instance rules.
6. Code review worked best when it asked, "What will the operator believe from this screen during a failure?"
