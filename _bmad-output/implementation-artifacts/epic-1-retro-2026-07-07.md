## Epic Summary

Epic 1 made EventStore domain modules more domain-centric. The platform now owns or hardens the canonical DomainService host, domain query routing, metadata propagation, generic read-model and cursor seams, projection and domain-event consumer seams, domain observability/Aspire defaults, Sample/Tenants adoption proof, and package/authoring guardrails.

Sprint status shows all seven Epic 1 stories marked `done`:

- 1.1 Canonical Domain-Service SDK Host
- 1.2 Domain Query Handler Routing
- 1.3 Generic Read Models And Query Cursors
- 1.4 Projection And Domain Event Consumer Seams
- 1.5 Domain Module Hosting Observability
- 1.6 Sample And Tenants Domain-Centric Adoption
- 1.7 DomainService Packaging And Guardrails

The retrospective also normalized the Epic 1 tracker state to `done` because all story keys are complete.

## What Went Well

- The canonical two-call DomainService host contract became test-proven instead of just documented.
- Query metadata moved from aspirational documentation to an end-to-end platform path: `QueryResult` through gateway/client metadata.
- Read-model and cursor work stayed additive and preserved public compatibility after review found constructor and deconstruction regressions.
- Projection and domain-event consumer seams gained marker-backed idempotency, support-safe validation, DAPR endpoint mapping tests, and explicit limits around handler atomicity.
- Observability moved into platform conventions: domain diagnostics registration, keyed lookup, state-store health checks, and DAPR app-health wiring.
- Sample became the clean minimal reference; Tenants became the realistic proof while preserving a documented transitional `/project` exception.
- Package and authoring governance became enforceable through manifest tests, guardrail tests, and active documentation alignment.

## Challenges And Growth Areas

- Several stories were oversized. Story 1.2, 1.3, 1.4, and 1.7 needed substantial review-driven rework because they touched public contracts, metadata semantics, or governance scans.
- Compatibility preservation was easy to under-specify. Reviews caught lost binary/source shapes, constructor naming drift, deconstruction shape drift, and metadata precedence contradictions.
- Fail-closed behavior needed repeated tightening: unknown freshness, invalid cursors, oversized cursors, stale cache freshness, DAPR marker release, and health endpoint policy all produced findings.
- Lightweight source scans are useful but brittle. Story 1.7 required multiple hardening passes and still has deferred gaps around cross-file/computed route values and broad Tenants host wiring.
- Tenants remains a transitional proof, not a fully clean domain module. It validates the platform path, but it still carries documented host composition exceptions.
- One story artifact was removed from HEAD after completion. Git history preserved Story 1.4 evidence, but current artifact discovery is less straightforward than the workflow expects.

## Review Pattern Synthesis

Every substantial Epic 1 story benefited from senior review:

- Story 1.1: method-aware route assertions and `/project` preservation behavior were tightened.
- Story 1.2: public query contract compatibility, freshness enforcement, cache-hit metadata, ETag precedence, and overflow-safe staleness checks were patched.
- Story 1.3: cursor handling, support-safe invalid-cursor responses, public compatibility, log redaction, and projection actor cursor rejection were hardened.
- Story 1.4: marker semantics, ULID validation, release behavior, skip completion, and DAPR marker tradeoffs were patched.
- Story 1.5: handler-only domain discovery, keyed diagnostics, ULID test data, blank-domain telemetry safety, and metric-side evidence were patched.
- Story 1.6: Tenants health response safety, root config binding tests, SDK query dispatch coverage, and source/package verification posture were improved.
- Story 1.7: manifest governance and domain-module guardrails required repeated hardening across package evaluation, state access detection, route parsing, string/comment handling, and project-shape checks.
