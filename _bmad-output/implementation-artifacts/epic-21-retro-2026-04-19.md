## 1. Epic Summary

**Trigger.** Fluent UI Blazor v4 reaches end-of-life in November 2026. v5 is a fundamental architecture change (FAST Web Components v2 → Fluent UI Web Components v3). Migration was executed proactively, with a **migration gate blocking all new UI work** until completion.

**Scope.** 949+ Fluent component occurrences across 62 `Admin.UI` razor files, 11 `Sample.BlazorUI` razor files, plus shared layout and tests.

**Approach.** Gate-first, regression-test baseline, staged mechanical renames ordered by compile dependency:

1. `21-0` bUnit baseline (14 smoke tests → regression signal)
2. `21-1` package bump + scoped-CSS bundling disabled
3. `21-2` layout & navigation foundation
4. `21-3` / `21-4` Appearance enum splits (Button, then Badge + Link)
5. `21-5` component renames
6. `21-6` dialog restructure (sync `@bind-Hidden` → async `ShowAsync/HideAsync` × 28 dialogs)
7. `21-7` toast API (`ShowSuccess` → `ShowSuccessAsync` × 106 call sites)
8. `21-8` CSS token migration
9. `21-9` DataGrid & remaining enum renames
10. `21-9.5` / `21-9.5.7` emergency: Admin.UI.Tests v5 migration, then runtime SSR-boundary fix
11. `21-10` Sample.BlazorUI alignment
12. `21-11` / `21-12` / `21-13` post-boot spin-offs (NavMenu CSS, FluentDesignTheme integration, UI bug batch)

**Final state.**

- Full solution compiles: **0 errors** (down from 253 after 21-1 package bump)
- Tier 1 non-UI tests: **753/753 green** throughout
- `Admin.UI.Tests`: **611/611 green** post-21-9.5.7
- `grep -E "(v4 API pattern)"` gates: **23 grep-for-zero gates passed** across 16 stories

## 2. What Went Well

1. **bUnit baseline before any v5 change (21-0)** gave an immediate before/after regression signal. Non-UI Tier 1 (753 tests) stayed green the entire epic — proof the v5 package upgrade itself was solid and all migration friction was correctly scoped to Blazor/UI semantics.
2. **Staged enum-rename strategy** (`21-3` ButtonAppearance, then `21-4` Badge+Link) split a monolithic Appearance change into two focused passes. Each story eliminated its own `CS0618` errors without exposing downstream projects to half-migrated state.
3. **Dialog restructure isolated as highest-risk story (`21-6`).** Allocated a dedicated session, explicitly flagged "highest regression risk." 28 dialogs converted from sync `@bind-Hidden` to async `ShowAsync/HideAsync` with zero runtime regressions on rapid open/close sequences.
4. **Toast API story (`21-7`) unblocked a cascade.** The 106 call sites of `ShowSuccess/ShowError/ShowWarning/ShowInfo` migrated to `*Async` extension helpers in a single pass. Dropped compile errors from 148 → 42, unblocking `21-8`, `21-9`, `21-10` runtime execution.
5. **Gate-first design worked.** The migration gate ("no new UI work until v5 complete") held for the full epic duration. No competing UI stories forced rework or merge conflicts.
6. **Post-boot discovery loop, not deferral.** When `21-9` boot exposed four pre-existing/latent bugs (NavMenu unstyled, theme toggle broken, Ctrl+K broken, TypeCatalog redirect loop), team pivoted immediately — scoped `21-11` / `21-12` / `21-13` on-the-fly rather than pushing them to a later epic. All four bugs closed in-epic.
7. **MCP migration server consulted before writing acceptance criteria.** `fluent-ui-blazor` MCP's `get_component_migration` surfaced one-to-many mappings (e.g., v4 `Appearance.Accent` + `Appearance.Neutral` both map to v5 `BadgeAppearance.Filled`) before implementation — caught what a naive mechanical rename would have missed.
8. **One-way commits per rename category.** Git history is clean: one story = one rename type = one reviewable diff. `git blame` tracks the rename source unambiguously.

## 3. What Struggled

1. **Downstream compile blockers daisy-chained visual verification gates.** Stories `21-4` (badge contrast), `21-6` (28 dialogs), `21-7` (toast rendering), `21-8` (CSS tokens), `21-9` (DataGrid alignment) all explicitly deferred visual ACs with "waiting for downstream stories to unblock Admin.UI compilation." Only after `21-10` + post-boot fixes did the full solution compile — by then, simultaneous visual sweeps for five stories' accumulated deferrals were impractical.
2. **CSS scoped-bundling disabled in `21-1`, re-included in `21-8`.** For seven intervening stories, `.razor.css` files were out of the build. Color tokens silently didn't apply, transitions silently didn't work. Problem wasn't visible until runtime.
3. **bUnit test infrastructure lagged production code.** `AdminUITestContext` depends on FluentUI being correctly imported and cascaded. Couldn't run Admin.UI tests until production compiled — broken from `21-1` all the way through to `21-9.5.7`. Tier 1 (non-UI) regressions were the only live signal for nine consecutive stories.
4. **SSR-boundary `@rendermode` on `<FluentProviders>` caused silent 500s.** Discovered first in `21-9.5.7` (`HostBootstrapTests` runtime failure), then discovered independently in `21-10` Sample.BlazorUI `App.razor`. v5 tightened `RenderFragment` serialization across interactive boundaries. Same bug hit twice — should have been a cross-project search-and-fix after first discovery.
5. **Deferred visual AC execution because Docker/DAPR unavailable in dev environment.** `21-10` Task 6 (3 pattern pages runtime verification, 28 dialog open/close, theme sweep) and Task 8.3 (Axe scan × 4 pages × 2 themes) explicitly punted to reviewer. Same for `21-11` hamburger drawer on smallest breakpoints.
6. **Seven raw-color inline style sites discovered late.** Health, Storage ×2, TypeCatalog ×2, TypeDetailPanel ×2 used `Style="color: #hex"` bypassing the Fluent 2 token system. `21-4` flagged the contrast concern; `21-8` moved most into CSS custom properties but not all. Design-system erosion was pre-existing, surfaced by v5 migration.
7. **CommandPalette Ctrl+K re-open flakiness (`21-13`)** — investigation capped at a scope-limited slot. Root cause (JS listener lifecycle × v5 dialog CloseMode × focus management interaction) not definitively nailed. Fix #1 partially resolved; may need post-launch monitoring.
8. **Theme mode strategy decided mid-epic, not up-front.** `21-2` punted on System mode, `21-12` spiked and re-confirmed it's out of scope without FluentDesignTheme rewire. Upfront decision would have saved two touch-points.

## 4. Recurring Review Feedback Themes

- **"Grep-for-zero verified, but visual regression not checked."** Gate pattern repeated across `21-4`, `21-6`, `21-7`, `21-8`, `21-9`. Mechanical correctness ≠ visual correctness.
- **Breaking changes manifest at runtime, not compile time.** V5 component structure changes (missing CSS classes, broken event handlers, DOM structure shifts) don't show up as `CS0618`. They appear only when the UI actually boots.
- **"Do NOT add new features, only migrate APIs."** Explicit discipline flag in `21-4` and `21-7` reviews. Team stayed disciplined — no QuickAction retry UX added to toasts, no Badge Shape/Size proactive additions. Good discipline, but repeated mention shows it was a recurring temptation.
- **Grep-for-zero happens after implementation, not before.** Manual verification at review time rather than CI gate.

## 8. Migration Methodology — What Worked, What to Change

**Worked:**

1. Baseline regression tests before any migration change.
2. Dependency-ordered compile gates (package → layout → enums → renames → dialogs → toast → CSS → DataGrid → sample).
3. Dedicated high-risk sessions (21-6 dialog restructure got its own session; paid off in zero runtime regressions).
4. MCP migration guides consulted at story-planning time, not implementation time.
5. Post-boot pivot: added three spin-off stories rather than deferring bugs.

**Change for next major framework migration:**

1. **Pair "disable" with "re-enable" in the same story.** `21-1` disabled scoped-CSS bundling; `21-8` re-enabled it. Seven silent-CSS stories in between. Keep the pair atomic.
2. **"Minimum viable boot" target earlier in the sequence.** Pick 1–2 core pages (Streams, Tenants) and drive them to visual-green *before* tackling the remaining 60+ pages. Frontloads runtime discovery; avoids the daisy-chained-deferral pattern.
3. **Automated grep-for-zero in pre-merge CI**, not manual at review time. If story spec says "zero `FluentButton Appearance=` occurrences remain," CI should enforce.
4. **Cross-project search on first discovery of a runtime pattern.** SSR-boundary `@rendermode` broke `HostBootstrapTests` in `21-9.5.7`, then broke Sample `App.razor` in `21-10` independently. Should have been one fix.
5. **Theme / design-system decisions upfront.** Theme mode (Light/Dark/System), raw-color audit, token strategy — decide before story 1, not mid-epic.
6. **Parallel streams for independent categories.** `21-3` (Button) and `21-4` (Badge) could have run in parallel once `21-1` and `21-2` landed. Conservative serial order cost calendar time without derisking anything.
7. **Runtime verification environment available to dev, not only to reviewer.** Docker / DAPR / AppHost must be available during story execution — deferring visual ACs to "reviewer will verify" creates an accountability gap.

## 14. Summary

Epic 21 is the textbook case for a proactive framework migration: triggered before EOL, gated to prevent merge chaos, baselined with regression tests, sequenced by compile dependency, and finished with zero v4 API references and both test tiers green. Cost: seven deferred visual-verification items (all logged, all closable post-launch), one runtime pattern (SSR-boundary) that we discovered twice instead of once, and eight post-boot bugs (four pre-existing, four migration-adjacent) absorbed into three spin-off stories.

The methodology transferable to future major migrations (Fluent UI v6, .NET 11, Aspire major bumps): gate-first, baseline-regression-test, pair-infrastructure-changes, "minimum viable boot" early, automate grep-for-zero in CI, runtime environment available during dev (not deferred to reviewer).
