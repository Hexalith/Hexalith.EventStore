## 3. What Went Well

1. **Scope discipline was strong.** Story 12.1 identified that the sample UI already existed and avoided unnecessary rewrites.
2. **The shared component design paid off.** Updating `CounterCommandForm.razor` applied success feedback consistently across all three pattern pages.
3. **The three refresh behaviors are distinct and teachable.** Notification, silent reload, and selective component refresh each demonstrate a different consumer strategy for projection change signals.
4. **Verification was explicit.** The story records cite component presence, command payload shape, disabled sending state, error handling, and SignalR subscription cleanup.
5. **Story 12.2 improved confidence.** It converted Story 12.1's deferred visual smoke test into recorded Tier 3 evidence.
6. **The UI respects product boundaries.** The sample presents HTTP command submission success/failure without pretending to show the full asynchronous command lifecycle.
7. **Old numbering confusion was called out early.** Story 12.1 documented which `12-*` artifact to ignore, reducing retrospective scope ambiguity.

---

## 4. What Struggled

1. **The epic was partly post-hoc verification.** Much of the implementation already existed, so the work was closer to audit and closure than greenfield feature delivery.
2. **Story 12.1 left visual verification to a reviewer.** That was later closed by Story 12.2, but the handoff created a temporary evidence gap.
3. **There is no automated Blazor component test layer.** The repository does not use bUnit for these Razor pages, so page behavior depends on code review and Tier 3 smoke testing.
4. **The status color scope needed careful wording.** The sample only observes success, failure, and idle states from command submission; full lifecycle colors are v2 dashboard scope.
5. **The epic depends on prior runtime infrastructure.** SignalR, projections, ETags, command API auth, AppHost wiring, and DAPR sidecars all have to work for the demo to feel real.
6. **Artifact numbering is still noisy.** Old `12-*` files can confuse later reviewers unless each retro is explicit about sprint-status scope.

## 7. Downstream Epic Preview - Epic 13

Epic 13 is **Documentation & Developer Onboarding**. Sprint status already marks its stories as `done`, so this retrospective is not preparing for an unstarted next epic. The relevant lesson is still useful: documentation should clearly explain what the sample UI proves and what it deliberately does not prove.

Epic 13 depends on Epic 12 in these ways:

1. Quick-start and onboarding material should point developers to the three refresh patterns as concrete examples.
2. Documentation should distinguish signal-only invalidation from authoritative query reloads.
3. Command feedback docs should avoid implying that HTTP acceptance equals final command lifecycle completion.
4. Developer guidance should identify when to use notification, silent reload, or selective refresh.
5. Runtime verification instructions should align with the Story 12.2 smoke-test path.

Alice (Product Owner): "The docs need to make the demo honest. The sample is strong because it teaches choices, not because it hides complexity."

Winston (Architect): "The next documentation layer should keep the architectural boundary visible: SignalR signals tell the UI to re-query; they are not the source of projection state."

## 11. Key Takeaways

1. **Verification can be delivery.** When useful code already exists, disciplined audit records are better than unnecessary rewrites.
2. **Shared components reduce UI drift.** One command form gave consistent behavior across the pattern pages.
3. **Runtime proof matters most for demos.** The sample UI is persuasive only when command submission and refresh behavior are observed together.
4. **Small UI signals need precise semantics.** Green submission feedback must not imply final command completion.
5. **Retrospectives must preserve scope boundaries.** Old-numbering artifacts and downstream completed epics can distort conclusions if not called out.
