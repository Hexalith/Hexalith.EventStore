## What Went Well

- The contract seam became explicit and scoped. `RestApiAttribute.Tag` and `RestRouteAttribute.ApiScope` normalization reduced accidental referenced-contract exposure.
- Generated REST controllers now have much stronger support-safety behavior around ETags, `Retry-After`, ProblemDetails, warning codes, validation errors, metadata headers, and documented status normalization.
- The Sample proof validated the intended host split: `Sample.Api` owns generated controllers, while the domain service and Blazor UI stay out of generated REST hosting.
- The Tenants proof exercised the harder case: source-mode AppHost wiring, generated Tenants API auth, system tenant scope, command/query identities, UI boundary guardrails, and package/source-mode validation.
- Story 2.5 preserved signal-only compatibility while adding scoped detail delivery, bounded metadata, DAPR direct/pubsub paths, reusable SignalR client APIs, and Redis backplane proof through the real client.
- Review findings were converted into tests repeatedly. The strongest examples were generated-controller runtime tests, authenticated in-process host tests, DAPR ACL parsing tests, and SignalR backplane evidence.

## Challenges And Growth Areas

- Generated REST support-safety was much larger than the initial story framing implied. Story 2.2 needed repeated review passes before HTTP status, ETag, retry, and ProblemDetails semantics were defensible.
- Generated API proof still lacks the reusable DAPR/Aspire smoke preflight promised by previous retrospectives. Sample and Tenants proofs are strong at compile, host, controller, auth, and handler-chain levels, but not at live generated API topology level.
- Some generator hardening remains outside Epic 2 scope: request-size limits, command status `Location` policy, `ArgumentException` problem text, invalid binding diagnostics, invalid tenant-source metadata, command rejection extension forwarding, and broader route/JSON-name diagnostics.
- External API host status-location semantics are unresolved. Generated command responses point to `/api/v1/commands/status/{id}`, but dedicated external API hosts do not necessarily expose that route directly.
- DAPR outbound routing headers need a clear replacement policy. Sample handlers currently append `dapr-app-id` and `dapr-api-token` rather than replacing conflicting values.
- One SignalR gap remains: raw hub leave calls validate scoped suffixes but not projection type or tenant id before group removal.

## Next Epic Preview

Epic 3 is `Release And Repository Reliability`. It builds directly on Epic 2 lessons:

- Story 3.1 re-tiers live-sidecar tests. Epic 2 proved that deterministic host/controller tests are useful, but live generated API and sidecar proof still need a separate lane.
- Story 3.2 hardens DAPR ETag timeout for integration conditions. Story 2.5 added ETag-before-broadcast behavior and real backplane evidence, so timeout/fail-open behavior matters for reliable test interpretation.
- Story 3.3 moves and verifies `references/` submodule layout. Epic 2 touched Tenants source-mode topology and reinforced the root-declared submodule boundary.
- Story 3.5 validates Debug source references and Release package references. Story 2.4 already showed that source/package-mode posture must be explicit for Tenants and Gateway dependencies.
- Story 3.6 makes release packaging manifest-driven. Epic 2 generated API host proofs should not leak submodule packages or source-mode assumptions into release output.

No PRD-level change is required for Epic 3. The existing Epic 3 plan already targets the reliability issues Epic 2 surfaced. The preparation risk is execution detail: Epic 3 stories should carry the generated API smoke preflight, package/source-mode validation, and integration evidence rules explicitly.

## Significant Discoveries

No significant discovery requires rewriting Epic 3. Epic 2 confirmed the current architecture:

- Generated REST controllers belong in dedicated external API hosts.
- The gateway remains the command/query policy boundary.
- Query and projection evidence must flow as platform metadata.
- Projection notifications are bounded freshness hints, not success proof.

The notable discovery is process-related: generated public protocols need runtime generated-host proof and adversarial support-safety review as first-class acceptance criteria, not after-the-fact hardening.
