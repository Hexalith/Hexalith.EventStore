## 3. What Went Well

1. **The six-layer security model became testable.** Epic 5 connected JWT validation, claims transformation, endpoint authorization, MediatR authorization, actor-level tenant validation, and DAPR access control into one traceable security story.

2. **Verification stories closed real gaps without broad rewrites.** Story 5.1 added 21 tests around OIDC/symmetric JWT configuration, structured auth logging, options validation, `alg:none` rejection, and claim fallback behavior.

3. **Authorization semantics are now explicit.** Story 5.2 documented case-sensitive tenant matching, case-insensitive domain and permission matching, open-by-default domain/permission behavior, actor-based validator configuration, and sanitized 403 responses.

4. **Tenant isolation is enforced across independent layers.** Story 5.3 verified actor IDs, state store keys, command status keys, pipeline keys, pub/sub topics, extension sanitization, metadata ownership, and payload redaction.

5. **DAPR policy expectations are now guarded by tests.** Story 5.4 added forward-looking checks for POST-only commandapi access, local-vs-production `defaultAction` divergence, and production-only operation grants.

6. **Keycloak E2E coverage is clearly separated from fast tests.** Story 5.5 verified that Keycloak-based E2E tests use real OIDC tokens, not synthetic JWTs, and are properly marked as E2E.

7. **Story records are much stronger than the previous epic.** Each Epic 5 story has completion notes, file lists or verification summaries, and explicit acceptance-criteria evidence.

---

## 4. What Struggled

1. **Epic 5 started with unresolved Epic 4 runtime-confidence debt.** R4-A3, R4-A4, R4-A5, and R4-A6 were not closed before Epic 5 evidence was recorded.

2. **Runtime security evidence still has environment boundaries.** Story 5.4 explicitly states NFR15 is fully met in production only because self-hosted DAPR lacks the same mTLS caller identity posture.

3. **Some story records still mix historical baselines.** Story 5.1 and Story 5.3 mention older baseline counts from previous story structures; the stories handled this correctly, but it makes trend reading harder.

4. **Story 5.2 did not record an exact test delta.** The story records the final passing counts and types of tests added, but not the count of newly added tests.

5. **The DAPR denial audit story remains partially accepted.** Story 5.4 accepted generic domain-service invocation failure logging because reliably distinguishing DAPR access-control denials was not straightforward with the current SDK surface.

6. **Keycloak E2E proves configured scenarios, not every future topology risk.** The tests prove the defined D11 security scenarios, but sidecar bypass, Kubernetes NetworkPolicy, and production mTLS posture remain deployment concerns outside the application test boundary.

## 10. Key Takeaways

1. **Security verification worked best where the team read existing code first.** The stories avoided broad rewrites and focused on targeted tests and evidence gaps.

2. **Layered security is only as strong as its weakest operational proof.** Unit/config coverage is strong, but pub/sub and live command-surface verification still matter.

3. **Real OIDC E2E tests are valuable because they prove what synthetic JWTs cannot.** Keycloak tests validate discovery, issuer, audience, JWKS, IdP claims, and the live auth pipeline.

4. **Local and production security posture must stay explicit.** Local DAPR policy tests are useful, but production enforcement depends on mTLS-backed identity.

5. **Epic 6 should observe security, not just operations.** Auth denials, tenant mismatches, DAPR policy failures, correlation IDs, and readiness status are all part of the security story.
