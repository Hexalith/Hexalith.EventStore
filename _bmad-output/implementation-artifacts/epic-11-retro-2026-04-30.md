## 3. What Went Well

1. **The epic was decomposed cleanly.** Contracts, event access, state storage, orchestration, discovery, and sample endpoint were delivered in a dependency-aware sequence.

2. **The public contract boundary is intentional.** Projection DTOs avoid leaking server-internal fields such as causation, user, metadata, extensions, and message identifiers.

3. **Event reading reused existing platform mechanics.** `AggregateActor.GetEventsAsync` follows the established metadata and batch state-read pattern, including missing-key and deserialization behavior.

4. **Projection state became queryable through the existing actor model.** `EventReplayProjectionActor` plugs into the established `ProjectionActor` type name and `CachingProjectionActor` cache path.

5. **Failure handling stayed aligned with command safety.** Projection update failures are logged and swallowed in the fire-and-forget path so command processing is not blocked by projection delivery problems.

6. **Post-review hardening improved the implementation.** Stories added guards for invalid projection state, notifier failure, DAPR actor serialization, malformed `/project` responses, null projection events, and tenant propagation.

7. **Configuration behavior is explicit.** Immediate mode runs the projection path, polling mode is gated and logged, and orphaned per-domain configuration is warned rather than made fatal.

8. **The sample endpoint is a useful reference.** The Counter projection handler demonstrates stateless replay, known event matching, unknown-event tolerance, and response shape compatibility.

---

## 4. What Struggled

1. **Full replay is a deliberate limitation.** The orchestrator currently calls `GetEventsAsync(0)`, so every trigger replays the aggregate history. This is simple and correct for the sample but not the final shape for heavier streams.

2. **Checkpoint tracking remains deferred.** The planning material names a checkpoint tracker, but Epic 11 does not implement last-successful-sequence persistence or incremental delivery.

3. **Polling mode is configured but not implemented.** `RefreshIntervalMs > 0` disables immediate delivery and logs that polling is deferred. Operators need to treat polling as unavailable until a poller exists.

4. **DAPR client testability constrained unit coverage.** `DaprClient.InvokeMethodAsync<TRequest,TResponse>` is non-virtual, so full invocation behavior had to move to integration paths rather than pure NSubstitute unit tests.

5. **End-to-end evidence is incomplete in the retrospective context.** Story 11.5 documents how to verify the complete Blazor/UI pipeline, but this retro did not run AppHost or capture a fresh UI proof.

6. **Projection semantics are distributed across several concepts.** EventPublisher, orchestrator, domain service resolver, aggregate actor, DAPR service invocation, projection actor, ETag actor, SignalR broadcaster, and query actor all participate. That increases the value of runtime traces.

7. **Previous retro continuity is missing for Epic 10.** No Epic 10 retrospective file was found, so there were no immediately prior action items to assess for follow-through.

## 8. Significant Discoveries

No Epic 11 discovery invalidates the downstream plan. The core architecture remains coherent.

Epic 11 did reveal preparation needs:

- Checkpoint tracking should become explicit product work rather than staying as a code comment.
- Polling configuration should either be backed by a poller or clearly constrained in operator/developer documentation.
- Full AppHost evidence should be captured for command, projection, query, ETag, SignalR, and UI refresh behavior.
- The DAPR service invocation test strategy should be made reusable so future cross-boundary stories do not rediscover the same limitation.
- Retro continuity needs attention because Epic 10 did not leave a retro record for SignalR lessons.

## 11. Key Takeaways

1. **Layered delivery worked.** Each story added one durable piece of the projection pipeline and carried forward prior-story intelligence.

2. **Fail-open behavior is essential for projections.** Command processing must not depend on projection delivery, but the operational story must make stale projection state visible.

3. **Correctness came before optimization.** Full replay reduced complexity, but the team should now turn checkpoint tracking into explicit follow-up work.

4. **Configuration must not overpromise.** Polling-mode config is currently a transparent deferral, not a complete feature.

5. **Cross-boundary features need running evidence.** Unit coverage is strong, but projection delivery crosses enough runtime components that AppHost proof is required for final confidence.
