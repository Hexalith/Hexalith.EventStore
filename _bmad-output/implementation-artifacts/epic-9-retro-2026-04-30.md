## 3. What Went Well

1. **Audit-first delivery worked.** Each story treated existing implementation as a hypothesis, checked it against acceptance criteria, and patched only real gaps.

2. **The 3-tier routing model is now pinned.** Entity, payload-checksum, and tenant-level routing are all verified with colon-separated actor IDs and checksum determinism.

3. **Self-routing ETags became robust.** Encode/decode behavior, quote handling, weak validator fail-open behavior, old-format migration, and non-ASCII projection types all received focused coverage.

4. **Projection invalidation has clear paths.** Direct and PubSub notification flows regenerate the ETag actor, and broadcast behavior is optional and fail-open.

5. **The query actor cache has meaningful safety rails.** Cache data is in-memory only, `JsonElement.Clone()` is called before caching, null ETags avoid unsafe caching, and deactivation naturally resets state.

6. **Runtime projection discovery was handled carefully.** First valid discovery wins, invalid values are rejected, and cross-domain projection mapping can work without putting `ProjectionType` on the client request.

7. **Response contract enforcement is honest about C# limits.** Non-null `ProjectionType` is compile-time enforced through NRT plus warnings-as-errors; non-empty semantics are runtime validated.

8. **Tests increased in the right places.** The added tests target migration, weak ETags, non-ASCII values, auto-notification, cache lifecycle, deactivation reset, and empty/whitespace `ProjectionType` handling.

---

## 4. What Struggled

1. **Planning text and implementation diverged.** The epics and PRD still describe hyphen-separated IDs and static TenantId metadata, while code intentionally uses colon separators and per-request tenant values.

2. **The API version decision remains implicit.** Story 9.1 kept query endpoints under `api/v1/queries`, while planning text still mentions `api/v2/queries`.

3. **Full HTTP ETag behavior is not fully proven.** Story 9.2 explicitly deferred the Tier 3 case where a valid self-routing `If-None-Match` returns 304 through the full HTTP pipeline.

4. **Latency targets are mostly architectural assertions.** Story records document the expected fast paths, but the evidence is primarily unit/integration slices rather than live measured topology evidence.

5. **Prior retro follow-through remains weak.** Several Epic 8 action items were not closed before or during Epic 9, especially story status drift, AppHost command-surface proof, Tier 3 triage, and release governance.

6. **Live topology was not available during this retrospective.** Aspire MCP detected no running AppHost, so this review is based on recorded story evidence and static artifact analysis.

7. **The epic depends on behavior that crosses component boundaries.** Query caching spans controllers, MediatR, actors, DAPR, domain services, and ETag actors; story-level slices can miss issues that only appear when the topology is running.

## 7. Downstream Epic Preview - Epic 10

Epic 10 is **SignalR Real-Time Notifications**. It builds directly on Epic 9's projection change notification and ETag regeneration chain.

Epic 10 depends on Epic 9 in these ways:

1. SignalR broadcasts must fire only after ETag regeneration succeeds.
2. SignalR group names must match ETag actor scope: `{projectionType}:{tenantId}`.
3. Clients must treat SignalR messages as invalidation signals and re-query through the ETag-backed query endpoint.
4. Redis backplane behavior depends on broadcasts being data-free and scoped correctly.
5. Reconnection behavior depends on the query endpoint remaining authoritative after missed signals.

**Current downstream status.** `sprint-status.yaml` already marks Epic 10 stories as `done`, so this retro is post-hoc relative to Epic 10 execution. Nothing discovered in Epic 9 invalidates the Epic 10 plan, but Epic 10 inherits Epic 9's need for end-to-end ETag/query proof.

## 8. Significant Discoveries

No Epic 9 discovery invalidates the downstream plan. The core architecture holds.

Epic 9 did reveal preparation needs:

- The full HTTP query pipeline needs valid self-routing ETag proof, including 304 and stale-ETag re-query behavior.
- Planning artifacts should be reconciled with implemented routing and API version decisions.
- ProjectionType enforcement should be documented as two-layer enforcement: compile-time non-null plus runtime non-empty validation.
- Live topology proof should be captured before claiming final confidence in query caching behavior.
- Prior retro follow-through needs stronger enforcement before more optional retrospectives are closed.

## 11. Key Takeaways

1. **Audit-plus-gap-fill was the right delivery shape.** Epic 9 avoided unnecessary rewrites and strengthened the exact weak spots.

2. **Caching correctness depends on small contracts being exact.** Separator rules, ProjectionType validation, ETag quoting, and fail-open behavior all matter.

3. **Focused tests are strong but not enough for cross-boundary behavior.** The query cache spans enough runtime components that Aspire evidence is still required.

4. **Planning drift is a real engineering risk.** Mismatched wording around hyphens, TenantId, ProjectionType, and route versioning can mislead future work.

5. **SignalR should remain downstream of query correctness.** Push notifications are invalidation hints; the query endpoint remains the source of truth.
