## 2. What Went Well

1. **Senior review (GPT-5.4 Copilot) consistently caught real issues the dev passed over.** 5/5 stories had reviewer-found HIGH or MEDIUM issues that became patches:
   - 2.1: `CommandRoutingIntegrationTests` was posting requests *without* `messageId` — failed at validation layer with 400 *before* reaching routing. Test was green only because it asserted the wrong outcome.
   - 2.2 HIGH: D1-1 placeholder (`AggregateType: "unknown"`) carried from Epic 1 retro — caught and fixed in `EventPersister` and `FakeEventPersister` together.
   - 2.2 MEDIUM: integration test asserted command acceptance + pub/sub counts only; didn't inspect persisted Redis state. Reviewer rewrote it.
   - 2.3 HIGH: snapshot path was forcing a *second* full event replay before domain-service invocation. Reviewer drove the `DomainServiceCurrentState` contract that eliminates it.
   - 2.4 HIGH: `CommandStatusController` validated correlation IDs as GUIDs, but the system uses ULIDs end-to-end. Real ULID inputs were 400ing.
   - 2.5: `conflictSource` extension was missing from the 409 ProblemDetails contract.
2. **D1-1 cleared incidentally.** The Epic 1 retro flagged `AggregateType: "unknown"` as the highest-severity carry-over (Pipeline-wide fix needed in Epic 2). Story 2.2's reviewer found it without it being scheduled, fixed it on the spot, and aligned the Fake to the real Persister's behavior. The standalone backlog story `post-epic-1-r1a1-aggregatetype-pipeline` is now duplicative — sprint-status should be reconciled.
3. **Story 2.3's design correction is a real architectural win, not just a fix.** Replacing "actor passes nothing → domain rehydrates from full replay" with "actor passes `DomainServiceCurrentState(snapshot + tail events)` → client rehydrates typed state" eliminates a second full replay on every command, keeps the server schema-ignorant, and works for both `EventStoreAggregate<TState>` and bare `DomainProcessorBase` consumers. The work also bounded `Task.WhenAll` in `EventStreamReader` (was unbounded across the full event range).
4. **Test ratchet held.** 651 → 652 (2.1 fix) → 656 (2.3 contract addition). No story dropped tests.
5. **Rule 12 (advisory writes never block the pipeline) was rigorously verified.** Every advisory write site in `AggregateActor` (Processing, EventsStored, EventsPublished, terminal Completed/Rejected/PublishFailed/TimedOut) is wrapped in try/catch with `OperationCanceledException` rethrow. Story 2.4 added the missing isolated test (`ProcessCommandAsync_AdvisoryStatusWriteFails_StillReturnsAccepted`) that proves the property at the actor seam, not just at the handler seam.
6. **The 5-step `AggregateActor` pipeline holds.** Verified live across the epic: idempotency → tenant validation (SEC-2) → rehydration → domain invocation → state-machine commit. Zero `DaprClient` state calls in the actor — `IActorStateManager` is the exclusive path (Rule 6 enforced by grep).
7. **`CommandStatus` and `CommandStatusRecord` 8-state model is internally consistent.** Integer assignments, terminal field shape (`EventCount`, `RejectionEventType`, `FailureReason`, `TimeoutDuration`), and TTL defaults all align across `DaprCommandStatusStore`, `InMemoryCommandStatusStore`, and the controller.
8. **Idempotency / concurrency separation is clean.** `IdempotencyChecker` (causation-keyed cache) and `ConcurrencyConflictException` (ETag failure) are independent paths with independent terminal states; `ConcurrencyConflictExceptionHandler` covers the cross-pipeline 409 surface and writes `Rejected` advisory status.

## 3. What Struggled

1. **"Verification story" framing kept producing surprises.** All five stories were nominally "audit existing code." Five times the reviewer found a HIGH or MEDIUM gap. Calling it "verification" undersells the work — the actual frame is "verify the contract, then expect to find one design or test gap per story." The story specs should budget review-found rework explicitly, not treat it as exceptional.
2. **Integration tests were systematically weak.** The 2.1 routing test posted requests without `messageId` and still passed (asserted the wrong thing). The 2.2 persistence integration test asserted command acceptance + pub/sub call counts, not Redis state. The pattern: integration tests treated success as "the call returned" instead of "the system reached the right end-state." Rewriting them to *inspect the state store* (Redis) is what made them load-bearing.
3. **Latent validation bugs survived to review.** Story 2.4's `CommandStatusController` rejected real ULID correlation IDs with GUID-only validation — a system-breaking bug that lived because no test fed it a real ULID. Also: 2.1's CausationId-vs-CorrelationId logging mismatch lived because the unit tests asserted the *wrong* relationship.
4. **Pre-existing out-of-scope failures are accumulating.** Story 2.1 documented 15 pre-existing Server.Tests failures: 4 `SubmitCommandHandler.Handle` NullRefs at line 81 (Pipeline scope), 1 validator extension-size test, 10 auth integration tests (Epic 5 scope). They're tracked but were not addressed in Epic 2. They stayed exactly the same number across all five stories.
5. **Epic 1 retro action follow-through is weak.** 5 carry-over items (R1-A1, R1-A2, R1-A4, R1-A6, R1-A7); 1 effectively addressed (R1-A1, by accident through reviewer catch). R1-A4 (`CommandStatus.IsTerminal()` extension) was supposed to fold into Story 2.4 — its status is uncertain in the verification notes. R1-A2 (`AssertTerminatableCompliance` helper), R1-A6 (`MissingApplyMethodException`), R1-A7 (Tier 2 tombstoning lifecycle test) are still in `backlog` as standalone stories. The standalone-story split was the correct response to "we need to address these before Epic 3 builds on them" — but Epic 2 didn't pick them up either.
6. **The DAPR SDK version drift between `CLAUDE.md` (1.17.0) and `Directory.Packages.props` (1.16.1) was noted in 2.1 dev notes and never resolved.** The .props file is the source of truth, but the doc disagrees. Cheap fix; keeps slipping.
7. **`AggregateType` was a 30-day-old TODO.** From a story-1.1 AI-Review MEDIUM in Epic 1 to a story-2.2 reviewer HIGH in Epic 2 — same bug, two epics. Found by reviewer, not by planning. The deferred-items table from Epic 1 retro was never re-read at Epic 2 planning.
8. **Bookkeeping drift inside stories.** Story 2.4's verification notes initially overcounted test coverage; the senior review's "MEDIUM bookkeeping" finding was specifically that the recorded counts didn't match the actual rerun. Small, but indicative — story records are written before the rerun completes and not always reconciled.

## 4. Recurring Review Themes

- **"It compiles and tests pass" ≠ "it works."** The CausationId test (2.1), the routing integration test (2.1), and the persistence integration test (2.2) were all green and all wrong. The compiler can't catch "test asserts wrong invariant" or "integration test never reaches the code under test."
- **Identity-format validation is a recurring trap.** 2.4 forced GUIDs where the system uses ULIDs. 2.1 flagged colon-rejection in `AggregateIdentity`. Identity formats are not interchangeable; treat `Guid.TryParse` on a correlation ID as an automatic FAIL.
- **Integration tests must inspect end-state.** Asserting "API returned 202" or "pub/sub was called once" is not an integration test. Asserting "Redis key `{tenant}:{domain}:{aggId}:events:1` contains an `EventEnvelope` with sequence 1" is.
- **Contract documentation drives reviewer comprehension.** Story 2.4's `Rejected` status was technically correct but ambiguous: domain rejection vs infrastructure failure both produced `Rejected`. The reviewer asked for explicit XML doc disambiguation (`RejectionEventType` set vs `FailureReason` set). Cheap fix, big clarity gain. Same pattern as Epic 1's `[DataMember]` discipline.
- **One absent method in a flow doesn't break a build.** D1-1 (`AggregateType` placeholder), R1-A4 (`IsTerminal()` extension folded-in), and the still-deferred R1-A2/R1-A6/R1-A7 all share this shape: missing-by-default, tests don't exercise them, only a real workload finds them.

## 9. Key Takeaways

1. **Senior review (GPT-5.4 Copilot) is a load-bearing pipeline stage in this project, not a rubber stamp.** 5/5 Epic 2 stories had reviewer-found patches; every patch was correct. Treat review-driven rework as the norm, budget for it in story specs.
2. **Integration tests must inspect end-state, not return values.** "Asserts the call returned 202" and "asserts pub/sub was called" are not integration tests — they're API smoke tests with extra steps. The 2.1 routing test and 2.2 persistence test were both green and both wrong until reviewers rewrote them to inspect Redis.
3. **Identity-format validation deserves an explicit story-spec rule.** `Guid.TryParse` on a correlation ID, message ID, or aggregate ID is an automatic FAIL — these IDs are ULIDs (or arbitrary strings, per `AggregateIdentity` validation). Fold into Epic 3's validation story specs (R2-A7).
4. **Deferred-items tables decay across epics.** D1-1 was a HIGH item from Epic 1 retro; nobody re-read the retro before Epic 2 planning; the reviewer caught it on Story 2.2 anyway. The catch worked, but it was lucky. R2-A9 makes the audit explicit.
5. **"Verification stories" are an underestimate.** Both Epic 1 and Epic 2 used the verify-and-complete shape and both surfaced real implementation gaps in every story. Plan accordingly: budget review-rework time, write story specs that *expect* gap discovery rather than asserting "code is already done."
6. **The right architectural fix is a contract, not a tweak.** Story 2.3's `DomainServiceCurrentState` payload (snapshot + tail in a typed DTO) was a contract addition that simultaneously fixed a perf issue (no second full replay), preserved server schema-ignorance, and worked for both consumer base classes. Cheap fixes ("add caching to the second replay") would have been worse.
7. **One missing extension method, helper, or exception type can carry across two epics.** R1-A2, R1-A4, R1-A6, R1-A7 are all "small, easy, latent." None broke Epic 2; all could break the first non-Counter aggregate or the first reactivated tombstoned actor. Schedule them (R2-A3, R2-A4) before that workload arrives.

## 12. Erratum (added 2026-04-26 by `sprint-change-proposal-2026-04-26-epic-2-retro-cleanup.md`)

**Affects:** §2 finding 2 ("D1-1 cleared incidentally"), §6 R2-A1, §10 commitments line "R1-A1 effectively cleared (Story 2.2)".

Its acceptance criteria must reject *both* `"unknown"` and `identity.Domain` as `AggregateType` values.

**Lesson.** Senior review catches surface symptoms reliably; semantic correctness still requires reading dev notes and prior decision records. R2-A9 (pre-Epic-3 audit of D1/D2 deferred items) explicitly mitigates this in the next epic.
