## 3. What Went Well

1. **Snapshots moved from baseline capability to tenant-aware control.** Story 7.1 added the missing tenant-domain interval tier while preserving domain and default fallbacks.

2. **The team handled a string-signature hazard explicitly.** Adding `tenantId` to `ShouldCreateSnapshotAsync` could have silently swapped string arguments. The story called this out and updated production, fake, and mock call sites together.

3. **Snapshot semantics stayed conservative.** Snapshot creation remains advisory, snapshots are an optimization rather than source of truth, and corrupt snapshot handling falls back to replay.

4. **Rate limiting built on the platform's intended primitives.** Story 7.2 extended the existing `AddRateLimiter` approach instead of introducing a second custom middleware path.

5. **Dynamic tenant configuration was designed to degrade gracefully.** DAPR config sync is optional, uses appsettings fallback, and avoids failing startup when the sidecar is unavailable.

6. **Per-consumer throttling was layered cleanly.** Story 7.3 used `PartitionedRateLimiter.CreateChained` so tenant and consumer protection remain independent concerns.

7. **ProblemDetails became more diagnostic.** Rate-limit responses now include `correlationId`, `tenantId`, and consumer context, helping clients and operators understand which protection triggered.

8. **The tests focused on the risky behavior.** New tests covered override resolution, validation rules, health endpoint exemptions, independent consumer buckets, and anonymous partition behavior.

---

## 4. What Struggled

1. **Story status drift appeared again.** Story 7.3's file header remains `Status: review` even though sprint-status marks it `done`.

2. **Epic 7 is broad for a compact epic.** Snapshots, dynamic tenant limits, and consumer throttling are cohesive under performance protection, but they touch different runtime planes and evidence types.

3. **Dynamic config proof is partly structural.** The story records rely on `IOptionsMonitor`, DAPR polling logic, and focused tests; they do not record a fresh end-to-end DAPR config-store update exercise.

4. **Rate limiter caching behavior creates explainability work.** Existing partitions keep prior limiter instances until they expire from idleness. This is acceptable, but it must be obvious to operators.

5. **Full Tier 3 evidence remains noisy.** Story records distinguish new-test success from known pre-existing failures, but the signal is still hard to read.

6. **Planning terminology does not perfectly match implementation terminology.** Epic text references tenant claim wording that differs from implemented `eventstore:tenant` usage, while consumer identity is consistently tied to `sub`.

7. **Older duplicate Epic 7 artifacts increase audit friction.** The current epic model and historical epic model both have `7-x` files, so future reviewers can easily inspect the wrong story records.

## 11. Key Takeaways

1. **Performance protection is a layered system.** Snapshots reduce backend pressure, tenant limits protect fairness, and consumer limits stop individual high-volume callers.

2. **Configuration patterns compound value.** The tenant-domain resolution pattern from snapshots directly informed rate-limit override design.

3. **Use platform primitives where they fit.** `PartitionedRateLimiter` and `CreateChained` gave the team a simpler and more maintainable path than custom middleware.

4. **Focused tests are necessary but not sufficient for runtime confidence.** Epic 7 has strong targeted evidence, but full-topology proof remains a separate obligation.

5. **Status hygiene matters.** When story files and sprint status disagree, retrospectives spend effort reconciling process state instead of learning from delivery.
