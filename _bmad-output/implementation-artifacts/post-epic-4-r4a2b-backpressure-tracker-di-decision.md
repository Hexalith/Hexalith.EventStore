I want the binary wire-or-delete decision for `IBackpressureTracker` (the defense-in-depth pipeline-level tracker shipped by Story 4.3 but never registered in production DI) made and executed in code,

### Wire-path ACs (apply only if AC C1 picks "wire")

W1. **`InMemoryBackpressureTracker` is registered in production DI as a Singleton.** A `services.TryAddSingleton<IBackpressureTracker, InMemoryBackpressureTracker>();` line is added to `src/Hexalith.EventStore.Server/Configuration/ServiceCollectionExtensions.cs` immediately after the existing `BackpressureOptions` registration block at lines 54-57 (HEAD `6430565`). **Use `TryAddSingleton`, not `AddSingleton`** — matches the existing project pattern at `:54` for `IValidateOptions<BackpressureOptions>` and is idempotent against re-registration during integration-test composition. Singleton lifetime is correct: `InMemoryBackpressureTracker` holds a `ConcurrentDictionary<string, int>` whose lifetime is per-instance and must outlive individual command requests. **Do NOT register as Scoped or Transient** — both lifetimes would create a fresh dictionary per request, defeating the entire point of in-flight tracking. Document the Singleton choice in an inline `// <reason>` comment if it improves readability; otherwise the lifetime choice is self-explanatory from the dictionary-state semantics.

W2. **`SubmitCommandHandler.Handle` calls `TryAcquire` BEFORE `WriteStatusAsync` and `WriteCommandAsync` (state-store-leak avoidance).** The handler is restructured so that immediately after `Log.CommandReceived(...)` (current `SubmitCommandHandler.cs:54` at HEAD `6430565`) and **BEFORE** the `statusStore.WriteStatusAsync` block (current line 58-77) and the `archiveStore.WriteCommandAsync` block (current line 79-91), a new block calls `_backpressureTracker.TryAcquire(aggregateActorId)`. **Why before-WriteStatus, not after-archive:** if `TryAcquire` ran after the writes (the original draft of this AC), a backpressure-rejected command would leave a `Received` status record AND an archived command body in storage with no terminal-state write — the rejection path never reaches `Completed` / `Rejected`. Under sustained backpressure that grows the status / archive stores unbounded with orphaned records (a state-store leak — exactly the failure mode R2-A6 was filed against). Putting `TryAcquire` before the writes means a rejected command never enters the status / archive stores at all.

   **Trade-off the dev MUST acknowledge in the Decision Record:** the HTTP 429 client receives the `correlationId` in the ProblemDetails body (per `BackpressureExceptionHandler`'s shape) but cannot poll `GET /commands/{correlationId}/status` afterward — there is no `Received` record to read. Document this in the Dev Notes as the "no-status-record-on-pre-actor-rejection" contract; this matches the behavior shape clients should already expect for `400`-class validation rejections (which also short-circuit before WriteStatus).

   **Canonical `aggregateActorId` shape (resolves A1).** The canonical actor ID is constructed by `CommandRouter.RouteCommandAsync` at `src/Hexalith.EventStore.Server/Commands/CommandRouter.cs:28-29` (HEAD `6430565`):
   ```csharp
   var identity = new AggregateIdentity(command.Tenant, command.Domain, command.AggregateId);
   string actorId = identity.ActorId;
   ```
   The `AggregateIdentity.ActorId` property is defined at `src/Hexalith.EventStore.Contracts/Identity/AggregateIdentity.cs:55` as `$"{TenantId}:{Domain}:{AggregateId}"`. **The wire-path implementation MUST construct the actor ID identically** — re-use the same `AggregateIdentity` instance / property, do NOT manually concatenate. Per `CommandRouter.cs:14-16` security comment: "Always derive actor ID from AggregateIdentity.ActorId. Never construct actor IDs via manual string concatenation. The chain of custody from CommandEnvelope through AggregateIdentity to actor ID must be unbroken." Manual concatenation in the pipeline tracker would silently key on a different string than the actor-state counter, breaking per-aggregate isolation.

   If `TryAcquire` returns `false`, throw `Hexalith.EventStore.Server.Commands.BackpressureExceededException` (the pipeline-side variant per the "two BackpressureExceededException classes" Dev Note in Story 4.3) with `CurrentDepth = backpressureTracker.GetCurrentDepth(aggregateActorId)`, `AggregateActorId = aggregateActorId`, `TenantId = request.Tenant`, `CorrelationId = request.CorrelationId`. The exception flows into the existing `BackpressureExceptionHandler` and produces the HTTP 429 + `Retry-After` response.

W3. **`SubmitCommandHandler.Handle` calls `Release` in a try/finally spanning `WriteStatusAsync` + `WriteCommandAsync` + `RouteCommandAsync` + post-routing telemetry.** If `TryAcquire` returned `true`, the slot MUST be released exactly once per the `IBackpressureTracker.Release` contract ("Must only be called after a successful `TryAcquire` call (returned true)"). Per W2's restructure, `TryAcquire` runs *before* the status/archive writes — the try/finally MUST therefore wrap the writes too, not just routing. Shape:
   ```csharp
   if (!_backpressureTracker.TryAcquire(aggregateActorId)) {
       throw new Hexalith.EventStore.Server.Commands.BackpressureExceededException(...);
   }
   try {
       // WriteStatusAsync, WriteCommandAsync, RouteCommandAsync, finalStatus read,
       // activityTracker / streamActivityTracker calls — current lines 58-154
   }
   finally {
       _backpressureTracker.Release(aggregateActorId);
   }
   ```
   The `Release` MUST run even if any inner call throws, including `OperationCanceledException` rethrow. **Do NOT release before tracker activities finish** — releasing too early leaks the slot before the command is fully tracked, defeating per-aggregate isolation under sustained load. **Do NOT release if `TryAcquire` returned `false`** — that contract violation is what the contract guards against. Exact-once is asserted by W6 Test 2 per M1's `Quantity.Exactly(1)` correction.

## Dev Notes

### Decision Framework: Wire vs Delete

The strongest evidence today (HEAD `6430565`) points toward **delete** as the default path. The reasons:

1. **The pipeline-layer tracker is currently a no-op in production code.** The 5-arg and 6-arg-with-tracker `SubmitCommandHandler` constructors at `:32-47` validate non-null on the parameter, then forward to the no-tracker primary at `:18-24`, discarding the tracker. `SubmitCommandHandler.Handle` does not call `_backpressureTracker.TryAcquire` or `_backpressureTracker.Release` because there is no `_backpressureTracker` field on the primary ctor — the parameter is throwaway. The XML doc on the structural-pin test at `SubmitCommandHandlerTests.cs:102-105` documents this: "the handler never invokes any method on the tracker — leaving it unconfigured documents that the stub is structural (non-null requirement only), not behavioural." The "tracker" exists in name only; the production hot path treats it as a no-op argument.
2. **The actor-level Step 2b check is the operational gate.** `AggregateActor.cs:184-234` reads the persisted `pending_command_count` actor-state key, fails open on state-read failure (intentional — availability > backpressure), compares against `BackpressureOptions.MaxPendingCommandsPerAggregate`, returns `CommandProcessingResult(BackpressureExceeded: true, ...)` on threshold reach. This path is wired, registered, tested 13 ways via `BackpressureTests.cs`, and is what produces the HTTP 429 response. Adding a pipeline-layer tracker on top of this provides marginal value — the actor activation cost is the only thing it avoids on a sustained-storm scenario.
3. **Per-instance in-memory state diverges from the persisted counter under multi-instance deployment.** `InMemoryBackpressureTracker` holds per-API-instance counters in a `ConcurrentDictionary`. In a multi-instance deployment (which is the normal Hexalith.EventStore deployment shape, per the AppHost / Aspire orchestration), each instance has its own counter; the actor's persisted counter is the authoritative cluster-wide value. The pipeline-layer tracker would either (a) reject commands the actor would have accepted (per-instance counter saturated while persisted counter is still under threshold — false positive) or (b) accept commands the actor would reject (per-instance counter not saturated while persisted counter is over — fast-fail bypass). Either way, the cluster-coherent behavior is owned by the actor; the pipeline tracker is at best a per-instance hint.
4. **Re-introduction is cheap if the operational need emerges.** The shape of the wire-path edits (W1–W7 above) is small: one DI registration, one `TryAcquire` call, one try/finally `Release`, one XML doc tweak, four new tests. If sustained-storm telemetry someday reveals that actor-activation cost is the dominant latency factor, re-introducing the tracker is a 30-line PR. Keeping dead code in the meantime so a future `Maybe` becomes a `Now` is reverse-cargo-cult engineering.

The strongest evidence for **wire**:

1. **The interface and implementation already exist with full unit-test coverage.** Throwing away 11 unit tests and 130-ish lines of working CAS-counter code feels wasteful, especially when the wiring is small. *Counter-argument:* the cost of dead code is not measured in lines or test counts; it's measured in confusion-load on every future reader who sees the constructor parameter and assumes it's wired ("Chesterton's fence" inverted — the fence is there but doesn't fence anything; readers waste cognitive cycles deducing that).
2. **Defense-in-depth is a generally-accepted pattern.** A pipeline-layer fast-fail before the actor activation has cost-savings under storm. *Counter-argument:* defense-in-depth needs to actually defend. A no-op parameter is not defense-in-depth; it is structural debt that *looks like* defense-in-depth to a fresh reader, which is worse than having no defense at all (because it gives a false sense of safety on review).
3. **An observed operational signal might be incoming (Trigger A).** If the dev who picks up R4-A2b is doing so because a backpressure-touching change is being authored that needs the pipeline-layer enforcement to be operational, **wire** is the correct path. *Counter-argument:* in the absence of such a signal, "we might need it later" is the textbook YAGNI pattern.

**Default: pick delete-removal unless Trigger A names a concrete operational signal.** This is consistent with the "removing dead code" CLAUDE.md guidance ("Don't add features, refactor, or introduce abstractions beyond what the task requires") and with the project's rule against dead-code retention. Document the choice and the trigger context in the Dev Notes per Task 1.3.

### Why the tracker ended up a no-op in production (5-Whys root cause from v3 elicitation)

### Latent `InMemoryBackpressureTracker._counters` dictionary-growth surface (out-of-scope observation)

Recorded only — NOT in scope to fix in R4-A2b. Surfaced by v3 elicitation pre-mortem (P3).

`InMemoryBackpressureTracker._counters.GetOrAdd(aggregateActorId, 0)` at `InMemoryBackpressureTracker.cs:34` adds the dictionary key on first read inside `TryAcquire`. For a typical Hexalith deployment this is small — backpressure rejection is the exception path.
