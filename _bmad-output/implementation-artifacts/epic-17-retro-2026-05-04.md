### Challenges and Friction

- System.CommandLine beta5 API drift appeared repeatedly. Stories hit changes such as `SetAction` replacing earlier patterns, read-only symbol names, missing `SetDefaultValue`, constructor differences, and changed option APIs.
- Console capture was fragile under parallel tests. Several stories had to avoid or isolate `Console.SetOut` / `Console.SetError` tests.
- Error semantics needed refinement as the surface expanded. 404-aware reads, tenant-not-found versus empty results, component filtering, file-write failures, HTTP auth failures, and timeout/cancellation behavior all needed explicit handling.
- Some unrelated pre-existing build issues surfaced during verification, especially integration-test `Program` ambiguity. The story records correctly separated those from Epic 17 scope.
- Story 17.7 generated a larger review surface than expected. The review found actionable issues around profile binding propagation, reserved profile names, completion scoping, and test expansion.

### Breakthrough Patterns

- `AdminApiClient` became the central extension point: `GetAsync`, `TryGetAsync`, `PostAsync`, `PutAsync`, and `DeleteAsync` gave every command a consistent HTTP and error-handling foundation.
- Output formatting became a reusable contract instead of per-command formatting code. The same JSON/CSV/table path supports human use and automation.
- Profile resolution formalized a four-layer priority model: explicit CLI values, environment variables, profile values, and defaults.
- The tool install smoke test closed the packaging gap. The CLI is not only buildable; it is installable, invokable, and releasable.

## 2. Next Epic Preparation

### Epic 18 Continuity Review

Epic 18 is already marked done in sprint-status, so this section reviews how Epic 17 prepared it rather than predicting future work.

Epic 18, Admin MCP Server, uses the same Admin API surface as Epic 17. The CLI lessons that mattered most for MCP were:

- Keep external admin interfaces thin HTTP clients over Admin API.
- Return machine-readable data consistently.
- Treat auth, not-found, service-unavailable, and unreachable cases as distinct operational signals.
- Keep stdout/protocol output clean. CLI separated stdout/stderr; MCP applies the stricter version of that rule by reserving stdout for JSON-RPC.
- Preserve testability through injectable clients and helper abstractions.

No Epic 17 discovery invalidated Epic 18's architecture. The direction stayed sound: MCP is a sibling consumer of Admin API, not a replacement for the CLI and not a DAPR-connected service.

### Key Takeaways

1. The shared Admin API strategy works. CLI and MCP can both remain thin clients without duplicating DAPR responsibilities.
2. Command-pattern consistency paid off. Once the first commands were hardened, later subcommands scaled predictably.
3. Tooling features need installed-path verification. Build success is not enough for a global tool.
4. Preview dependencies require explicit upgrade discipline.
5. Retrospective continuity needs care. Epic 16 has no saved retro, so Epic 17 had no prior action ledger to close.
