## 2. What Went Well

1. **Audit-and-complete framing kept blast radius bounded.** Treating each story as "verify, gap-fix, regression-protect" instead of greenfield was the right call for a regenerated epic on top of mature code. No story rewrote what was already working.
2. **Named-argument discipline as policy, not afterthought.** Both 1.1 and 1.2 dev notes called out positional-record swap-vulnerability up front and mandated `new EventMetadata(MessageId: …, AggregateId: …, …)` at every construction site. Zero swap bugs across ~100 sites — *because the rule was hardcoded in the spec*, not because anyone remembered.
3. **`Hexalith.Commons.UniqueIds` adopted cleanly.** Story 1.3 brought in the first external Contracts NuGet dependency without creating a custom `UlidId` value object. Kept Contracts thin (D12).
4. **Tombstoning design rigour (1.5).** Two rounds of advanced elicitation surfaced the runtime-only `Apply(AggregateTerminated)` obligation, snapshot/replay safety, and irreversibility semantics *before* code was written. Documented in dev notes, not lost.
5. **Counter sample as proof rig.** Every contract change rode the Counter aggregate through its tests — validated the developer-facing surface as it changed.
6. **Test-count ratchet caught regressions.** 628 → 634 → 648 → 651 baseline tracking gave a quick "did we drop tests?" check at story boundaries.
7. **Story 1.4's 30+ subtasks for an XS story.** Verbose-on-purpose to defeat LLM "audit theater." Worked — agent verified each of 10 public types exhaustively rather than spot-checking.

## 3. What Struggled

1. **Massive cross-project blast radius for positional-record changes.** Story 1.1 hit 30+ files; 1.2 hit 60+. Pure grep + manual edit, no tooling. Tedious and error-prone — protected only by tests + named arguments.
2. **`[DataMember]` is a hidden runtime contract.** Story 1.2 spec called out the CorrelationId pattern → MessageId rule, but it's not enforced by the compiler. Without the explicit doc, `MessageId` would have silently vanished during DAPR actor state serialization. Easy trap for any future positional-record field on serialized records.
3. **`AggregateType: "unknown"` hardcoded placeholder still open.** `EventPersister.cs:82` and `FakeEventPersister.cs:60` (Story 1.1 AI-Review MEDIUM item) — left as deferred follow-up. Carried 6 weeks. Needs Epic 2 architectural fix to thread aggregate type through the pipeline.
4. **Code reviewer false-positive CRITICAL on 1.5.** GPT-5.4 Copilot missed `CommandStatusTests.cs:39` (the terminal-status test already existed) and demanded a "fix." Round-2 elicitation caught the misread. Wasted cycles — the lesson is that CRITICAL findings need a verification command (e.g., `grep -n "Terminal" CommandStatusTests.cs`) so the dev can confirm before accepting a status change.
5. **`Apply(AggregateTerminated)` is a runtime-only constraint.** Documented in `<remarks>`, but a domain dev implementing `ITerminatable` for a new aggregate will pass close-works/first-rejection tests and only fail on actor reactivation. Future stuck-actor risk if no automated check ships.
6. **Test-baseline drift between stories.** Story 1.4 dev notes say "baseline 634 from 1.2" but actual count by 1.4 close was 651. Manual baselines in story specs go stale across an epic.
7. **XML doc enforcement only enabled on Client.csproj** (Story 1.4 fix). Contracts, Server, SignalR, Aspire, Testing still don't fail builds on missing XML docs. UX-DR19 ("XML documentation on all public types") is honour-system in those packages.

## 4. Recurring Review Themes

- **Mechanical correctness ≠ semantic correctness.** Reviewers caught `Guid.NewGuid()` used where `UniqueIdHelper.GenerateSortableUniqueStringId()` was required (D12), and `tenant:domain:id` composite passed where a bare `aggregateId` was expected. Compile-green was a weak signal across the epic.
- **UTF-8 vs char count.** Story 1.4 review fixed a `charCount * 2` byte-budget calc in `SubmitCommandRequestValidator`. Latent — only shows on multi-byte payloads. Reviewers found what tests didn't.
- **XML doc gaps surface only when build enforces them.** Enabling `GenerateDocumentationFile=true` on Client revealed zero gaps — but only because the docs were already there. Without enforcement, future drift is invisible.

## 9. Key Takeaways

1. **For positional records consumed by serialization, hardcode the named-argument rule and the `[DataMember]` requirement in the story spec.** Compiler won't catch either.
2. **"Audit-and-complete" stories need verbose subtask lists when executed by LLM dev agents** — ambiguity invites audit theater.
3. **Runtime-only constraints (like `Apply(AggregateTerminated)`) deserve a Testing-package compliance helper**, not just a `<remarks>` warning.
4. **CRITICAL review findings need a verification command.** False-positive CRITICALs are expensive — a `grep` line costs reviewers nothing and saves dev cycles.
5. **Enable XML doc generation per package as you ship public surface.** Cheap, prevents drift, catches what reviewers won't.
