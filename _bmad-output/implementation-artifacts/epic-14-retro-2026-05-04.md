## 1. Epic Summary

Epic 14 established the shared administration backend for Hexalith.EventStore. It created the contract package, DAPR-backed service implementations, REST controllers with role and tenant authorization, a dedicated Admin.Server host wired into Aspire, and OpenAPI/Swagger support for the admin surface.

**Sprint-status scope.**

| Story | Title | Outcome |
|---|---|---|
| 14.1 | Admin Abstractions - Service Interfaces and DTOs | Created a pure `Admin.Abstractions` contract package with 10 service interfaces, immutable DTOs, redaction behavior, and 107 focused tests |
| 14.2 | Admin.Server - DAPR-Backed Service Implementations | Created 10 service implementations, options, state-store key helpers, DI registration, and 65 focused Admin.Server tests |
| 14.3 | Admin.Server - REST API Controllers with JWT Auth | Added 7 admin controllers, authorization policies, claims transformation, tenant filter, request DTOs, and middleware-path integration tests |
| 14.4 | Admin.Server - Aspire Resource Integration | Added `Admin.Server.Host`, AppHost/Aspire wiring, DAPR access control updates, host tests, and real JWT bearer verification follow-up |
| 14.5 | Admin API OpenAPI Spec and Swagger UI | Added admin OpenAPI transformers, Swagger UI gating, OpenAPI tests, host tests, and co-hosting path-prefix guards |

**Artifact note.** The implementation folder also contains same-prefix files such as `14-1-docker-compose-deployment-guide-and-configuration.md` through `14-8-disaster-recovery-procedure.md`. This retrospective follows `sprint-status.yaml` and treats those as adjacent deployment/documentation artifacts, not the five tracked Epic 14 admin API stories.

## 3. What Went Well

1. **Contract-first foundation held.** Story 14.1 created a pure `Admin.Abstractions` package with no DAPR, ASP.NET Core, or MediatR dependency leakage.
2. **Admin writes stayed behind the correct boundary.** Story 14.2 kept write operations delegated to CommandApi rather than writing directly to the state store.
3. **Security semantics became explicit.** Story 14.3 added ReadOnly, Operator, and Admin policy hierarchy plus tenant filtering.
4. **Testing became more realistic.** Story 14.3 moved from direct controller-only confidence to TestServer with custom auth headers, and Story 14.4 later strengthened host tests with real JWT bearer tokens.
5. **Aspire topology gained a real admin resource.** Story 14.4 added `admin-server` to the AppHost and DAPR access-control model.
6. **OpenAPI had co-hosting protection.** Story 14.5 added path-prefix guards so admin transformers do not mutate CommandApi operations.
7. **Review follow-up was effective.** Story 14.2 and 14.4 review notes show concrete hardening around dependencies, pagination, status-code mapping, auth context placement, sidecar access, and real authorization outcomes.

---

## 4. What Struggled

1. **Story status drift reappeared.** `sprint-status.yaml` marks Story 14.3 and Story 14.5 `done`, but both story files still say `Status: review`.
2. **Framework reality changed implementation shape.** Story 14.3 needed FrameworkReference cleanup, CA1062 fixes, TestServer instead of WebApplicationFactory, and header-based test auth because ThreadStatic claims did not flow.
3. **DaprClient testability required adaptation.** Story 14.2 found non-virtual `InvokeMethodAsync<TRequest,TResponse>` could not be mocked with NSubstitute and refactored to the abstract request-based invocation path.
4. **Actor-state access limits shaped the admin design.** Story 14.2 had to distinguish admin-index/state-store reads from actor-state access that requires CommandApi mediation.
5. **OpenAPI typed schema quality is not fully closed.** Story 14.5 records deferred `ActionResult<T>` migration because `IActionResult` plus `[ProducesResponseType]` was enough for current generation but weaker as a long-term contract.
6. **Some Swagger/API polish was skipped.** Admin example transformers, admin error reference endpoints, and related tests were skipped from Story 14.5 based on priority tiers.
7. **Full solution validation had a known unrelated blocker in some story records.** Story 14.1, 14.2, and 14.4 cite a pre-existing IntegrationTests `Program` ambiguity, while focused admin and Tier 1 validations passed.

## 5. Story Analysis Themes

**Common struggles.**

- Authorization confidence required tests through the ASP.NET Core middleware path, not just controller instantiation.
- Central package management and warnings-as-errors exposed small dependency and analyzer mismatches quickly.
- Admin API work repeatedly had to defend boundaries: no direct writes, no payload leakage, no transformer bleed into CommandApi, and no tenant bypass.
- The implementation artifact folder still contains duplicate same-prefix story numbers, so retrospectives need sprint-status scoping.

**Breakthrough moments.**

- Story 14.1 set a clean contract layer that later stories could consume without infrastructure dependency creep.
- Story 14.2 converted DAPR limitations into explicit design rules: actor state through CommandApi, admin indexes through state-store reads, write operations through CommandApi.
- Story 14.3 made role hierarchy and tenant isolation executable through tests instead of relying on attribute inspection.
- Story 14.4 turned the admin backend into a first-class Aspire resource with its own sidecar and access policy.
- Story 14.5 recognized OpenAPI as a downstream contract for CLI and MCP consumers, not just Swagger UI decoration.

**Testing and quality insights.**

- Story 14.1 recorded 107 Admin.Abstractions tests plus Tier 1 validation.
- Story 14.2 recorded 65 focused Admin.Server tests covering DI, options, key derivation, request construction, bearer-token forwarding, error mapping, null handling, and pagination.
- Story 14.3 recorded 67 new tests and 878 Tier 1 tests passing with no regressions.
- Story 14.4 recorded 10 new host tests and 1,059 Tier 1 tests passing with no regressions.
- Story 14.5 recorded 177 Admin.Server tests, 15 Host tests, and Tier 1 coverage passing with no regressions.
- Runtime Aspire confidence is represented through story records, but this retrospective did not rerun Aspire.

## 11. Key Takeaways

1. A shared admin contract prevents Web UI, CLI, and MCP from creating separate versions of the same product.
2. Admin write operations must preserve the command pipeline boundary.
3. Authorization tests are only meaningful when they exercise middleware, policies, claims transformation, and tenant filtering together.
4. OpenAPI is a contract for downstream tooling, so schema quality and path isolation matter.
5. Sprint-status scope must be stated when implementation artifacts contain duplicate numbering.
