## Intent

**Problem:** Tenants already has a dedicated generated REST API host, but Story 2.4 needs the EventStore root topology and Tenants tests to prove that host is the only external tenant-management REST surface and that it preserves gateway authorization, status, freshness, and archive behavior for both commands and queries.

**Approach:** Harden the existing `Hexalith.Tenants.Api` adoption instead of rescaffolding it: wire it into the EventStore AppHost source-mode topology, align its gateway-sidecar behavior with the Sample API host, add command/query runtime proof through the compiled generated routes, and add structural guardrails that keep the Tenants domain service and UI out of the generated REST host role.

## Boundaries & Constraints

**Always:** Keep `Hexalith.Tenants.Api` as the only generated external tenant-management API host; generated actions must delegate through `IEventStoreGatewayClient`; Tenants domain identifiers remain caller-supplied strings, not ULIDs/GUIDs; EventStore envelope ids generated by the REST generator remain ULIDs; the API host uses `RestTenantSource.System` and forwards the validated caller bearer to EventStore; UI hosts use Tenants/EventStore client seams directly.

**Block If:** Removing the Tenants domain service's generic imported EventStore command controller surface is required for compatibility but not clearly covered by existing tests; the implementation would require changing REST generator public semantics, query/command contract names, generated route templates, or Tenants product authorization behavior.

**Never:** Do not add generated or hand-written per-message MVC command/query controllers to `Hexalith.Tenants`, `Hexalith.Tenants.UI`, or EventStore UI hosts; do not make the API host reference the Tenants domain implementation or UI host; do not parse cursors, ETags, JWT payloads, or bearer tokens; do not broaden DAPR service invocation beyond EventStore gateway command/query POST operations.

## I/O & Edge-Case Matrix

| Scenario | Input / State | Expected Output / Behavior | Error Handling |
|----------|--------------|---------------------------|----------------|
| Tenants generated query | Authenticated `GET /api/tenants`, `/api/tenants/{tenantId}`, `/api/users/{userId}/tenants`, or `/api/global-administrators` receives a gateway success with metadata | Returns raw payload and forwards bounded `ETag` plus present `X-Hexalith-*` metadata; gateway request uses tenant `system`, declared domain/query/projection identity, and route/query payload values | Missing metadata omits headers without fabricating freshness, paging, ETag, or projection evidence |
| Tenants generated query `304` | Authenticated generated query forwards `If-None-Match` and gateway returns not-modified with a strong ETag | Returns empty `304`, preserves strong `ETag`, and forwards only present support-safe metadata headers | Missing or weak ETag follows generated safe `502` problem behavior from Story 2.2 |
| Tenants generated command | Authenticated tenant/global-admin command route receives a non-null body whose route values match body values | Calls `SubmitCommandAsync` once with tenant `system`, declared domain, aggregate id, command type, serialized payload, generated ULID message id, and returns `202` with command status `Location` and `Retry-After` | Null body or route/body mismatch returns safe `400 application/problem+json` before gateway call |
| Unauthorized caller | Generated Tenants API route is called without a bearer token | ASP.NET auth returns `401` and no gateway command/query is submitted | No resource existence, raw payload, or internal metadata is disclosed |
| Boundary regression | Domain service or UI gains `[assembly: RestApi]`, REST generator analyzer reference, controller mapping, per-message controllers, or API-host project references | Guardrail tests fail before runtime | Generic domain-service compatibility endpoints may remain only as explicitly documented platform imports, not as generated Tenants REST routes |

</intent-contract>
