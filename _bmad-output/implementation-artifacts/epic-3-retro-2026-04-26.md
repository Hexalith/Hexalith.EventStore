## 3. What Went Well

1. **The API contract became coherent.** `POST /commands`, `GET /commands/status/{correlationId}`, replay, validation, auth, concurrency, infrastructure failures, rate limiting, OpenAPI, and problem documentation now use the same vocabulary and response model.

2. **Error response standardization paid off.** Story 3.2 introduced `ValidationProblemDetailsFactory`; Story 3.5 generalized the pattern with `ProblemTypeUris` and handler updates. This reduced URI drift and made review easier.

3. **The ULID lesson from Epic 2 was applied in status-query stories.** Story 3.3 avoided the previous GUID-only status-query bug and documented the D5 exception for status queries that do not have a single tenant context.

4. **Replay traceability improved.** Story 3.4 added `OriginalCorrelationId` so operators can link a replay attempt to the original failed command without relying only on logs.

5. **Security posture improved at the client boundary.** Story 3.5 removed internal terms and implementation detail from 409 and 503 responses, removed pre-pipeline correlation IDs from 401/503 bodies, and added a DAPR sidecar unavailability handler before the global 500 handler.

6. **OpenAPI moved beyond "document exists."** Story 3.6 added grouped endpoints, a working Counter command example, command lifecycle remarks, and `/problems/{errorType}` pages backed by tests.

---

## 4. What Struggled

1. **Previous retro actions were not fully closed before Epic 3.** Several Epic 1 and Epic 2 cleanup stories remain in backlog while Epic 3 is already marked done.

2. **Story records and sprint status drifted.** `sprint-status.yaml` marks `3-6-openapi-specification-and-swagger-ui` as `done`, but the story file still says `Status: review`.

3. **The replay endpoint still violates the Epic 3 ULID rule.** `src/Hexalith.EventStore/Controllers/ReplayController.cs` still uses `Guid.TryParse(correlationId)` and returns "not a valid GUID format." This conflicts with the Epic 3 amendment that inbound correlation IDs must not be GUID-validated.

4. **The `AggregateType` semantic bug is still present in testing fake behavior.** `src/Hexalith.EventStore.Testing/Fakes/FakeEventPersister.cs` still writes `AggregateType: identity.Domain`, which matches the Epic 2 erratum's warned-against wrong value.

5. **`CommandStatus.IsTerminal()` was not elevated to a shared contract helper.** `CommandStatusController` still owns a private `IsTerminalStatus` helper. The Epic 2 action asked for a shared extension used by status writers and tested centrally.

6. **Tier 3 and live-system confidence remained uneven.** Multiple story records document Docker/Aspire or Tier 3 exclusions. The final story used Tier 2 WebApplicationFactory tests, which is useful, but not a substitute for validating the running Aspire topology.

---

## 10. Key Takeaways

1. **Verification stories are implementation stories with an audit-first entry point.** Every "already implemented" area still needed contract fixes, tests, or documentation.

2. **Error contracts benefit from central primitives.** `ProblemTypeUris` and shared ProblemDetails factories were the right abstraction level.

3. **Prior retrospectives need a gate, not a memory.** The Epic 2 retro identified critical prep items, but several stayed open while Epic 3 closed.

4. **Identifier semantics must be enforced mechanically.** A single remaining `Guid.TryParse` in replay is enough to violate the platform's ULID contract.

5. **Test fakes are part of the product surface.** A fake that encodes the wrong aggregate-type semantics can create false confidence in downstream packages.
