### Functional Requirements — by section

#### Command Processing (FR1–FR8, FR49) — 9 FRs
- **FR1** Submit command via REST (messageId ULID, aggregateId ULID, commandType kebab `{domain}-{command}-v{ver}`, payload)
- **FR2** Validate structural completeness (required fields, ULID format, MessageType kebab, well-formed JSON)
- **FR3** Route command to correct aggregate actor by `tenant:domain:aggregate-id`
- **FR4** Receive correlation ID on submission (defaults to messageId; messageId is idempotency key)
- **FR5** Query command status by correlation ID
- **FR6** Operator replay of failed command via Command API
- **FR7** Reject duplicate commands on optimistic concurrency conflict
- **FR8** Route failed commands to dead-letter topic with full context
- **FR49** Detect/reject duplicates by tracking processed message IDs per aggregate (idempotent success on re-submit)

#### Event Management (FR9–FR16, FR65, FR66) — 10 FRs
- **FR9** Append-only immutable event store (events never modified or deleted)
- **FR10** Strictly ordered, gapless sequence numbers per aggregate stream; cross-aggregate ordering not guaranteed
- **FR11** 14-field metadata envelope, two-document storage (metadata JSON + payload JSON, D14)
- **FR12** Reconstruct aggregate state by replay from sequence 1 to current
- **FR13** Snapshots at admin-configured intervals (default 100); EventStore signals threshold, domain service produces snapshot inline
- **FR14** Reconstruct from snapshot + subsequent events == full replay
- **FR15** Composite key (`tenant:domain:aggregateId`) isolation
- **FR16** Atomic event writes — 0 or N events, never partial subset
- **FR65** `metadataVersion` field for envelope schema evolution
- **FR66** Aggregate tombstoning (terminal event); rejects subsequent commands; stream remains immutable

#### Event Distribution (FR17–FR20, FR67) — 5 FRs
- **FR17** Pub/sub publish using CloudEvents 1.0
- **FR18** At-least-once delivery
- **FR19** Per-tenant-per-domain topics (`{tenant}.{domain}.events`)
- **FR20** Continue persisting when pub/sub temporarily down; drain backlog on recovery
- **FR67** Per-aggregate backpressure (default 100 pending) → HTTP 429 with Retry-After

#### Domain Service Integration (FR21–FR25) — 5 FRs
- **FR21** Pure function `(Command, CurrentState?) -> DomainResult` (aggregate type + event types + payloads only)
- **FR22** Convention-based routing (DAPR AppId == domain name, method "process"); overridable via appsettings.json or DAPR config store
- **FR23** EventStore invokes domain service, enriches events with all 14 metadata fields
- **FR24** Process commands for ≥2 independent domains in same instance
- **FR25** Process commands for ≥2 tenants per domain with isolated streams

#### Identity & Multi-Tenancy (FR26–FR29) — 4 FRs
- **FR26** Derive actor IDs / stream keys / topics from `tenant:domain:aggregate-id` (tenant from JWT D15, domain from message type prefix D13)
- **FR27** Data-path isolation across tenants
- **FR28** Storage key isolation
- **FR29** Pub/sub topic isolation

#### Security & Authorization (FR30–FR34) — 5 FRs
- **FR30** JWT authentication on Command API
- **FR31** Authorize on tenant + domain + command type claims
- **FR32** Reject unauthorized at API gateway pre-pipeline
- **FR33** Tenant verification at actor level
- **FR34** Service-to-service via DAPR access control policies

#### Observability & Operations (FR35–FR39) — 5 FRs

- **FR37** Trace failed command from dead-letter back to originating request

#### Developer Experience & Deployment (FR40–FR48) — 9 FRs

#### Query Pipeline & Projection Caching (FR50–FR64) — 15 FRs
- **FR50** 3-tier query actor routing: EntityId | Checksum | empty
- **FR51** ETag actor per `{ProjectionType}:{TenantId}` storing self-routing ETag
- **FR52** `NotifyProjectionChanged(projectionType, tenantId, entityId?)` helper (DAPR pub/sub default, or service invocation)
- **FR53** Endpoint pre-check decodes self-routing ETag → call ETag actor → 304 if match (no query actor activation)
- **FR54** Query actor as in-memory page cache; learns projection type mapping from `IQueryResponse<T>` on cold call
- **FR55** SignalR signal-only "changed" broadcast on ETag regeneration; group = ETag actor ID
- **FR56** SignalR hub hosted in EventStore server with Redis backplane
- **FR57** Query contract NuGet defines mandatory metadata (Domain, QueryType, TenantId) + optional EntityId; ProjectionType NOT on consumer side
- **FR58** Coarse invalidation (per projection per tenant) — design-simplicity trade-off
- **FR59** SignalR client helper auto-rejoins groups on connection recovery
- **FR60** ≥3 reference patterns for handling SignalR "changed" signal in Blazor (toast / silent reload / selective refresh)
- **FR61** ETag format `{base64url(projectionType)}.{guid}` wrapped per RFC 7232; undecodable → cache miss
- **FR62** `IQueryResponse<T>` enforces non-empty `ProjectionType` at compile time
- **FR63** Query actor learns projection type mapping at runtime; resets on actor deactivation
- **FR64** Doc guidance: short projection type names for compact ETags

### Additional requirements / constraints captured in PRD

- **Stack pin**: .NET 10 LTS, DAPR 1.16.1, Aspire 13.x, C# 14
- **Architecture decisions referenced**: D12 (ULID via Hexalith.Commons.UniqueIds), D13 (`{domain}-{name}-v{ver}` message type), D14 (two-document storage), D15 (server-derived tenant/userId/causationId), D16 (4+1 client command), D17
- **Versioning**: Monorepo single-version SemVer 2.0; envelope changes are MAJOR (irreversible)
- **JSON-only serialization in v1** — two-document storage enables future binary serialization without envelope schema changes
- **Domain == DDD bounded context, 1:1 with domain microservice**
- **Triple-layer multi-tenant isolation**: actor identity, DAPR policies, command metadata
- **6-layer authn/authz defense-in-depth**: JWT validation → claims extraction → endpoint authorization → MediatR pipeline behavior → actor identity verification → DAPR policy enforcement
- **Three-tier testing strategy**: unit (no DAPR) / integration (DAPR test container) / contract (full Aspire)
- **Snapshot policy controlled by EventStore (WHEN), domain service controls WHAT** (inline during command processing per FR13)
- **Phasing classification**: v1 = pipeline + query/projection caching + SignalR ; v2 = Admin tooling (UI/CLI/MCP) + Workflows + Sagas; v3 = enterprise (GDPR, versioning, HA failover, tiered storage); v4 = ecosystem (templates, gRPC, plugins)
- **Tenant lifecycle is delegated to Hexalith.Tenants peer service**, not implemented inside EventStore admin
- **Irreversible decisions** (must validate before GA): event envelope schema, message type convention, identity tuple
